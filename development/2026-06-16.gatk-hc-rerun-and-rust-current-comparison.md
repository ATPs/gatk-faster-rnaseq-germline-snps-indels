# Same-input GATK HaplotypeCaller rerun vs current Rust caller

## Purpose

This records the Module 6 audit requested after repeated Rust/GATK mismatch rounds. The key question was whether the previous differences came from other pipeline stages or from the Rust HaplotypeCaller replacement itself.

The answer is clear from this run: the same-input Java GATK rerun is exactly identical to the existing GATK baseline VCF by exact allele key and genotype. The current Rust caller mismatch is from the Rust caller algorithm, not from upstream pipeline drift.

## Interpretation

The existing baseline and the same-input GATK rerun are identical:

- same raw and filtered record counts;
- 100% exact allele-key match for raw and filtered VCFs;
- 100% genotype match on shared records;
- identical PASS/non-PASS and filter-count distributions.

Therefore the Rust mismatch is not caused by a different upstream BAM, interval list, reference, dbSNP, or VariantFiltration command. It is caused by the Rust caller algorithm.

The current Rust caller remains a fast pileup SNP/short-indel caller, not a GATK-equivalent HaplotypeCaller:

- it misses 11,407 of 36,435 GATK PASS calls;
- it shares only 742 of 4,400 GATK PASS indel/complex calls;
- it emits 3,322 Rust-private PASS calls;
- even when the allele key is shared, 9,962 calls are homozygous alternate in GATK but heterozygous in Rust.

The current overlap-correction changes did not move the result closer to GATK. They reduced Rust-private PASS calls compared with the earlier speed-first run, but they also reduced GATK PASS sensitivity from the earlier `89.80%` snapshot to `68.692%`.

## Required direction

More pileup threshold tuning is unlikely to reach GATK-equivalent output. The Rust implementation needs to mimic the Java core documented in `development/2026-06-16.gatk-haplotypecaller-core-algorithm.md`:

1. active-region detection and smoothing;
2. GATK-style region finalization, clipping, and overlapping mate quality correction;
3. local read-threading assembly with reference and non-reference haplotypes;
4. haplotype-to-reference Smith-Waterman alignment and event maps;
5. PairHMM read-vs-haplotype likelihoods;
6. allele likelihood marginalization from haplotypes to variant events;
7. GATK-like diploid genotyping, allele subsetting, QUAL/GQ/PL calculation, and annotations used by downstream filters.

Until those pieces are implemented, `rust_haplotype_caller` should be treated as a fast development workbench, not a drop-in replacement for Java GATK `HaplotypeCaller`.
