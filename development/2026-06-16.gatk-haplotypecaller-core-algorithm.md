## Source files and entry points

Local GATK source is under `/data/p/gatk/gatk/src`.

Important Java files:

- `HaplotypeCaller.java`: tool wrapper, default filters, read transformer, traversal integration.
- `HaplotypeCallerEngine.java`: `isActive()` and `callRegion()` core control flow.
- `AssemblyBasedCallerUtils.java`: read finalization, clipping, overlapping mate correction, local assembly setup, event and allele mapping helpers.
- `ReadThreadingAssembler.java`: kmer graph construction, graph pruning, dangling branch recovery, best haplotype extraction, and haplotype-to-reference alignment.
- `PairHMMLikelihoodCalculationEngine.java`: read-vs-haplotype likelihood matrix using PairHMM.
- `HaplotypeCallerGenotypingEngine.java`: event extraction from haplotypes, allele likelihood marginalization, genotype likelihoods, genotyping, and annotation.
- `GenotypingEngine.java`: allele frequency model, call confidence, LOW_QUAL decision, allele subsetting.
- `FragmentUtils.java` and `FragmentCollection.java`: overlapping paired-fragment quality correction.

## Java HaplotypeCaller flow in this mode

1. Read filtering and base transformation:
   - `HaplotypeCaller.getDefaultReadFilters()` delegates to `HaplotypeCallerEngine.makeStandardHCReadFilters()`.
   - Standard filters are mapping quality >= 20, mapping quality available, mapped, not secondary, not duplicate, vendor QC pass, non-zero reference length alignment, good CIGAR, and wellformed read.
   - `makeStandardHCReadTransformer()` applies `IUPACReadTransformer(true)`.

2. Active-region discovery:
   - `HaplotypeCallerEngine.isActive()` works per locus using the base pileup and reference base.
   - It calls `ReferenceConfidenceModel.calcGenotypeLikelihoodsOfRefVsAny(...)` with `hcArgs.minBaseQualityScore` and `activeRegionAltMultiplier`.
   - A minimal genotyper converts ref-vs-any likelihoods into an active probability. AssemblyRegion traversal later groups/smooths active loci into active regions.
   - This is not equivalent to calling every pileup position independently.

3. Region finalization before assembly:
   - `callRegion()` exits early for inactive/no-read regions via `referenceModelForNoVariation`.
   - For active regions, `AssemblyBasedCallerUtils.assembleReads(...)` first calls `finalizeRegion(...)`.
   - With `--dont-use-soft-clipped-bases`, soft-clipped bases are hard-clipped before assembly.
   - Low-quality tails are clipped using `minTailQuality = minBaseQualityScore - 1`.
   - Adapter sequence and active-region padding clipping are applied.
   - Reads shorter than the minimum usable length after trimming are later removed before genotyping.

4. Overlapping mate correction:
   - `finalizeRegion()` sorts reads and calls `cleanOverlappingReadPairs(...)` when `doNotCorrectOverlappingBaseQualities` is false.
   - `FragmentCollection.create(...)` finds overlapping read pairs by read name and coordinates.
   - `FragmentUtils.adjustQualsOfOverlappingPairedFragments(...)` keeps both reads, but adjusts qualities in the overlap:
     - same base in both mates: both base qualities are capped to half the default PCR SNV quality, default Q20;
     - conflicting bases with `setConflictingToZero=true`: both base qualities are set to zero;
     - if indel-quality caps are supplied, insertion/deletion qualities are capped in the overlap.
   - GATK does not collapse an overlapping fragment into one observation at this stage.

5. Local assembly and haplotype generation:
   - `AssemblyBasedCallerUtils.assembleReads(...)` creates a padded reference haplotype with 500 bp assembly padding.
   - It uses `ReadThreadingAssembler.runLocalAssembly(...)`, not a pileup-only caller.
   - `ReadThreadingAssembler` builds kmer graphs, prunes chains, can recover dangling branches, finds best haplotype paths, and aligns each non-reference haplotype back to the reference with Smith-Waterman.
   - The output is an `AssemblyResultSet` containing the reference haplotype, candidate non-reference haplotypes, event maps, padded reference span, and region for genotyping.

6. Event collection, trimming, and pileup event injection:
   - `callRegion()` combines assembled events, good pileup events, forced alleles, and removes bad pileup events.
   - `AssemblyRegionTrimmer.trim(...)` narrows the active region to the variant-containing span.
   - `assemblyResult.injectPileupEvents(...)` can add artificial haplotypes for good pileup events, but this supplements assembly; it is not the whole caller.

7. PairHMM likelihood calculation:
   - Reads are split by sample over the region for genotyping.
   - `PairHMMLikelihoodCalculationEngine.computeReadLikelihoods(...)` evaluates every read against every candidate haplotype.
   - PairHMM uses base qualities, insertion/deletion qualities, gap continuation penalties, PCR error model, optional Dragstr scores, and caps read base qualities by mapping quality unless disabled.
   - This produces `AlleleLikelihoods<GATKRead, Haplotype>`, a read-haplotype likelihood matrix.

8. Allele likelihood marginalization and genotyping:
   - `HaplotypeCallerGenotypingEngine.assignGenotypeLikelihoods(...)` builds event maps for haplotypes and iterates event start positions.
   - It converts active haplotypes to variant contexts with `getVariantsFromActiveHaplotypes(...)`.
   - It creates an allele-to-haplotype map with `createAlleleMapper(...)`, including spanning deletion handling.
   - Haplotype likelihoods are marginalized to allele likelihoods for each event.
   - `calculateGLsForThisEvent(...)` builds genotype likelihoods.
   - `resolveGenotypePriorCalculator(...)` uses Hardy-Weinberg priors from SNP and indel heterozygosity unless Dragstr priors are active.
   - `GenotypingEngine.calculateGenotypes(...)` runs the allele frequency model, applies `standard-min-confidence-threshold-for-calling`, subsets output alleles, assigns filters such as LOW_QUAL, and creates final genotypes.

9. Annotation:
   - `makeAnnotatedCall(...)` runs the annotation engine over the final call.
   - For this RNA hard-filter pipeline, `FS` and `QD` are especially important because downstream `VariantFiltration` filters on them.
   - `FS` is derived from allele-aware read likelihoods and strand counts, not only raw pileup counts.
   - `QD` is `QUAL` normalized by depth with GATK-specific capping for unusually high QD values.
   - Other standard annotations such as RMS mapping quality can be finalized before output.

## Current Rust caller gap checklist

Current Rust module 6 code is in `src/rust/src/haplotype_caller.rs` with CLI `src/rust/src/bin/rust_haplotype_caller.rs`. It is useful as a fast workbench but still differs from the Java algorithm in major ways:

| GATK core component | Current Rust status | Impact |
|---|---|---|
| Standard read filters | Mostly approximated: MQ >= 20, mapped, non-secondary, non-duplicate, vendor QC pass, reference-consuming CIGAR | Needs exact GOOD_CIGAR / Wellformed / IUPAC review |
| Active-region model | Not implemented; Rust scans requested pileup positions/windows directly | Changes which loci are assembled/called |
| Region finalization/clipping | Not equivalent; no full `finalizeRegion()` behavior | Soft clips, low-quality tails, adapters, and region padding can differ |
| Overlapping mate correction | Partially adjusted in source, but must be tested against `FragmentUtils` | Can create or remove private calls |
| Local read-threading assembly | Not implemented | Major cause of SNP/indel discordance |
| Haplotype-to-reference SW alignment | Not implemented | Indel representation and nearby variants differ |
| PairHMM read-haplotype likelihoods | Not implemented | Genotype likelihoods and QUAL are not GATK-equivalent |
| Allele likelihood marginalization | Not implemented | Multi-allelic, spanning deletion, and nearby-event behavior differ |
| AF/genotyping engine | Simplified diploid model only | QUAL, PL, GQ, and allele output differ |
| Annotation engine | Simplified FS/QD/DB only | Hard-filter behavior can diverge |

The biggest expected mismatch is indel behavior. GATK discovers and genotypes indels through active-region assembly and PairHMM likelihoods; a pileup-only indel caller cannot match it closely.

## Same-input rerun and comparison plan

Before additional Rust algorithm changes:

1. Rerun Java GATK `HaplotypeCaller` on the exact same baseline recalibrated BAM, interval list, reference, and dbSNP.
2. Run the same `VariantFiltration` step on the rerun raw VCF.
3. Compare existing baseline filtered VCF versus the GATK rerun filtered VCF. If these differ materially, use the rerun as the immediate comparison baseline for Rust.
4. Compare Rust filtered VCF versus the GATK rerun filtered VCF using exact allele keys (`CHROM`, `POS`, `REF`, `ALT`), both all-record and PASS-only.
5. Split discordance into SNP and indel categories.
6. Only after this same-input Java baseline is confirmed should Rust code be changed further.

Recommended output directory:

`/XCLabServer002_fastIO/gatk-faster-rnaseq/SRR949115_broad_hg38_run2/module6_gatk_hc_rerun_20260616`

Recommended acceptance target for the Rust replacement remains:

- at least `99.9%` PASS exact allele-key concordance against the same-input Java GATK result;
- Rust-private PASS calls below `0.5%` of GATK PASS calls unless explicitly reviewed and accepted;
- a substantial wall-time speedup after matching the algorithmic behavior.
