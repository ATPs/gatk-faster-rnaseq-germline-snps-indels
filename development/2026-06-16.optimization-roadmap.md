# RNA-seq germline variant pipeline optimization roadmap

## Aim

Define the next optimization roadmap after the first baseline and `sambamba markdup` benchmark on `SRR949115`. The goal is to make the RNA-seq germline SNP/indel pipeline substantially faster while keeping output close enough to the current GATK pipeline to remain useful.

The user preference for the next phase is:

- speed first;
- do not pursue a shared BQSR model across many samples for now;
- direct pipeline parameter changes have only limited upside;
- start module-level Rust rewrites for the measured bottlenecks, one tool at a time.

## Current state

The current runnable pipeline is `gatk-faster-rnaseq-germline-snps-indels/src/run_pipeline.py`.

Measured on `SRR949115` with Broad hg38 resources:

- baseline total: `3412.767 s` (`56.879 min`);
- `sambamba` estimated end-to-end total: `4275.519 s` (`71.259 min`).

The first `sambamba` attempt did not improve end-to-end runtime:

- `MarkDuplicates` itself was only slightly faster;
- downstream `ApplyBQSR` became much slower;
- output VCFs remained very similar, so the slowdown is not explained by a major content change.

Current major bottlenecks by wall time are:

1. `HaplotypeCaller`: `1425.558 s`
2. `SplitNCigarReads`: `968.256 s`
3. `BaseRecalibrator` + `ApplyBQSR`: `642.034 s` in baseline

Additional observations from inspection:

- the current exon BED contains `1,668,627` rows and the interval list contains `1,671,994` rows, which is too fragmented for efficient interval traversal;
- `HaplotypeCaller` spent about `60.9 s` in PairHMM during the baseline run, so most of its wall time is outside the native PairHMM kernel;
- the slow `sambamba` `ApplyBQSR` run stalled for a long time around `chr8:73,291,xxx`, so any decision about `sambamba` must be based on repeated runs, not a single measurement;
- `SplitNCigarReads` is not a trivial CIGAR split: it also performs the STAR `MQ 255 -> 60` transform by default, two-pass overhang fixing, supplementary-tag repair, mate-cigar repair, and tag cleanup.

## Optimization principles

1. Keep the standard GATK pipeline as the reference implementation.
2. Each Rust replacement must first be a standalone CLI that can be run by its existing `step_*.py` wrapper.
3. Rewrite one module at a time and only connect it to `run_pipeline.py` after step-level validation.
4. Prefer narrow replacements with clear contracts over rewriting a whole stage at once.
5. Keep a compatibility mode when exact behavior matters, and allow a separate speed-first mode only when the downstream VCF impact is measured.
6. Treat BQSR as per-sample for now. A generic cross-sample recalibration model is a research experiment, not the default pipeline.
7. Do not fuse adjacent stages until each standalone replacement has passed acceptance tests.

## Roadmap

### Phase 1: make the benchmark harness decision-grade

Before more optimization work, make sure timing comparisons are reproducible.

Tasks:

- run long jobs in `tmux` under the normal shell environment from `AGENTS.md`;
- repeat key benchmark modes at least 2 times before drawing conclusions;
- record for every run:
  - `timings.tsv`;
  - total wall time;
  - VCF counts;
  - `bcftools isec` comparison to baseline;
  - `samtools flagstat` and selected `samtools stats`;
- keep all large files under `/XCLabServer002_fastIO`.

Reason:

The current `sambamba` result may include a run-specific pathology in `ApplyBQSR`. One run is not enough to choose the next architecture.

### Phase 2: create the Rust module harness

Before replacing biological logic, create a minimal Rust workspace under `src/rust/` and keep Python as the pipeline orchestrator.

Deliverables:

- `src/rust/Cargo.toml` with one binary per replacement module;
- a shared BAM/VCF/reference utility crate if it removes real duplication;
- Python step wrappers that can choose `--backend gatk` or `--backend rust`;
- small synthetic fixtures for CIGAR, qualities, known-sites masking, and duplicate edge cases;
- a comparison script for BAM, recal table, and VCF summaries.

Acceptance:

- `cargo test --release` passes under the normal zsh + conda base environment;
- every Rust binary supports `--help` and exits non-zero on invalid required input;
- every Python wrapper still defaults to the current GATK backend;
- `python -m py_compile src/*.py` still passes.

### Phase 3: Rust module backlog and order

The initial order should be:

1. `rust_interval_tools`
2. `rust_apply_bqsr`
3. `rust_split_n_cigar_reads`
4. `rust_base_recalibrator`
5. `rust_mark_duplicates`
6. caller-side Rust experiments for `HaplotypeCaller`

This order gives quick feedback first, then attacks the large preprocessing costs, and leaves the caller rewrite until the validation surface is manageable.

#### Module 1: `rust_interval_tools`

Purpose:

- replace slow GATK `BedToIntervalList` and `SplitIntervals` usage for exon interval preparation;
- emit merged BED and balanced `.interval_list` shards directly from exon BED + reference `.dict`;
- unblock repeatable `HaplotypeCaller` scatter experiments.

Scope:

- parse BED or interval_list;
- sort and merge intervals by reference dictionary order;
- split into N shards by total covered bases, not by row count;
- preserve GATK-compatible interval_list headers.

Acceptance:

- output interval_list files are accepted by GATK `HaplotypeCaller -L`;
- total covered bases match the input after merge rules are applied;
- no interval has invalid contig, start > end, or coordinates outside the reference;
- shard covered bases are within `10%` of the mean unless a single interval is larger than that limit;
- interval preparation wall time is at least `5x` faster than GATK `BedToIntervalList` + `SplitIntervals` on the current exon set.

#### Module 2: `rust_apply_bqsr`

Purpose:

- replace GATK `ApplyBQSR`, using the recal table produced by GATK `BaseRecalibrator`;
- reduce a narrow per-read BAM transformation before rewriting recal-table generation.

Scope for v1:

- support the current pipeline's substitution-quality path;
- read GATK recal tables;
- preserve coordinate order, header, read groups, flags, mate fields, and required tags;
- rewrite base qualities in the same domain used by the current GATK command;
- produce a coordinate-sorted BAM and `.bai`.

Acceptance:

- synthetic reads with known qualities produce exact expected recalibrated qualities;
- `samtools quickcheck` passes on output BAM and index;
- `samtools flagstat` total, mapped, duplicate, paired, properly paired, secondary, and supplementary counts match GATK `ApplyBQSR`;
- on `SRR949115.split.bam`, per-base quality differences versus GATK are either exact or explained by a documented rounding rule;
- downstream filtered VCF has at least `99.9%` PASS concordance against GATK `ApplyBQSR` output before replacing the default backend;
- step wall time is at least `2x` faster than baseline GATK `ApplyBQSR` or it is not worth adopting.

#### Module 3: `rust_split_n_cigar_reads`

Purpose:

- replace GATK `SplitNCigarReads`, the largest RNA-specific preprocessing cost;
- make a fast mode available while keeping a compatibility path for strict comparison.

Scope for v1 fast mode:

- split reads at `N` CIGAR operations;
- apply the STAR mapping-quality transform `MQ 255 -> 60`;
- preserve read order as much as possible for coordinate-sorted output;
- remove or repair tags that are invalid after splitting;
- write sorted BAM and `.bai`.

Scope for v2 compatibility mode:

- implement overhang clipping/fixing behavior;
- handle supplementary alignments and mate-cigar repair to match GATK more closely;
- preserve the default RNA-seq read filters used by the current GATK path.

Acceptance:

- synthetic CIGAR tests cover `N`, `S`, `H`, insertions, deletions, supplementary reads, paired reads, unmapped mates, and reads near exon boundaries;
- v2 compatibility mode matches GATK `SplitNCigarReads` BAM-level behavior on selected chromosome windows, except for documented harmless header or ordering differences;
- v1 fast mode may differ, but downstream filtered VCF must keep at least `99.5%` PASS concordance and private PASS calls must stay below `0.5%` of baseline PASS calls;
- full `SRR949115` step wall time is at least `2x` faster than GATK `SplitNCigarReads` for adoption.

#### Module 4: `rust_base_recalibrator`

Purpose:

- replace `BaseRecalibrator` after `rust_apply_bqsr` is already validated;
- compute the recal table faster while keeping known-sites masking and covariates compatible with the current pipeline.

Scope:

- implement the current substitution covariates first: read group, reported quality, machine cycle, and sequence context;
- use the same known-sites VCFs as the baseline;
- keep the same RNA-seq read filters unless a measured experiment justifies changing them;
- output a GATK-compatible recal table that `rust_apply_bqsr` and GATK `ApplyBQSR` can both consume.

Acceptance:

- synthetic known-sites tests prove that bases overlapping dbSNP/known-indel records are excluded from error counting;
- recal table event counts match GATK for small deterministic windows;
- empirical quality and estimated recalibration values match GATK within a documented numeric tolerance;
- applying the Rust table with GATK `ApplyBQSR` produces downstream VCF concordance at least as good as applying the GATK table with `rust_apply_bqsr`;
- step wall time is at least `2x` faster than GATK `BaseRecalibrator` for adoption.

Cross-sample BQSR note:

- BQSR is strongly tied to sample, read group, lane, chemistry, cycle, and context. For 1,000 similar files, the default should still be per-sample or per-read-group recalibration.
- A shared model can be tested only as a separate speed-first experiment after per-sample Rust BQSR works. It must be rejected if it changes VCF PASS concordance, quality distributions, or Ti/Tv-like sanity metrics beyond the thresholds in this document.

#### Module 5: `rust_mark_duplicates`

Purpose:

- revisit duplicate marking only after the larger bottlenecks improve;
- provide a controlled alternative to Picard/GATK and `sambamba`.

Scope:

- duplicate marking for coordinate-sorted RNA-seq BAM without UMI support in v1;
- preserve Picard-compatible duplicate metrics;
- keep UMI-aware duplicate marking out of scope unless the input data actually has UMI structure.

Acceptance:

- duplicate flags and duplicate metrics match Picard on synthetic paired-end cases;
- `samtools flagstat` duplicate counts match Picard on `SRR949115`;
- downstream VCF concordance is at least as good as the previous `sambamba` comparison;
- end-to-end runtime improves, not just duplicate-marking runtime.

Fusion decision:

- Do not merge `MarkDuplicates` and `SplitNCigarReads` in the first Rust implementation. Duplicate marking is defined on the original aligned records, while `SplitNCigarReads` changes record structure.
- After both standalone modules pass, a fused streaming implementation can be evaluated only as an I/O optimization. Its acceptance target is identical duplicate flags, identical split output semantics, and lower combined wall time than running the two Rust modules separately.

#### Module 6: caller-side Rust experiments for `HaplotypeCaller`

Purpose:

- avoid a full caller rewrite until preprocessing and scatter/gather are no longer dominant;
- investigate smaller Rust pieces only if `HaplotypeCaller` remains the main wall-time cost after the earlier modules.

Allowed experiments:

- active-region profiling on recalibrated BAMs;
- interval/window prefiltering that reduces regions passed to GATK `HaplotypeCaller`;
- lightweight pileup summaries for deciding where full local assembly is needed.

Out of scope for now:

- a production Rust replacement for full GATK `HaplotypeCaller`;
- rewriting local assembly, graph pruning, genotyping, and annotation in the same phase.

Acceptance for any caller-side filter:

- no malformed or unsorted VCF output after gather;
- PASS sensitivity against baseline filtered VCF is at least `99.9%`;
- missed baseline PASS calls must be reviewed by region and consequence before adoption;
- wall-time reduction must be large enough to justify the biological risk, target at least `2x` faster caller stage.

## Validation standard

For every optimization candidate, record:

- end-to-end wall time;
- per-step wall time;
- output BAM size and index status;
- VCF total count;
- PASS count;
- non-PASS count;
- `bcftools isec` shared/private counts against baseline.

Acceptance levels:

- exact-flow changes:
  - output should be identical or near-identical after normalization and sorting;
- speed-first biological or implementation changes:
  - target PASS concordance of at least `99.5%`;
  - target private PASS count no more than `0.5%` of baseline PASS calls;
  - no malformed BAM/VCF, no missing indexes, and no header/contig incompatibility.

## Immediate next tasks

The next concrete work items should be:

1. Create `src/rust/` as the Rust workspace and add a minimal smoke binary.
2. Add shared comparison utilities for BAM summary, recal-table summary, and VCF concordance.
3. Implement `rust_interval_tools` first because it is small, low-risk, and unblocks faster caller scatter tests.
4. Add optional `--backend gatk|rust` switches only to the affected step wrappers, while keeping GATK as the default.
5. Validate `rust_interval_tools` on the existing exon BED and Broad hg38 dictionary.
6. Start `rust_apply_bqsr` after the harness and interval tool are accepted.
7. Record every module result in a new `development/YYYY-MM-DD.<module>-validation.md` file before moving to the next module.
