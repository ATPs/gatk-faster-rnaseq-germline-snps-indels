# HaplotypeCaller assembly-read subset consistency

## Aim

Remove the production/replay mismatch in assembly-read selection.

Before this change:

- production `scan_call_window` computed a Java-like per-start `use_for_assembly` mask, but then ignored it and assembled from `read_bases_list.iter().take(500)`;
- replay `scan_replay_window` assembled from the masked subset instead.

That made selected-region replay less trustworthy as a production proxy and biased production assembly toward BAM input order.

## Code changes

Changed file:

- `src/rust/src/haplotype_caller.rs`

Implementation:

- Added a shared helper `select_assembly_reads(...)`.
- Changed production assembly-read selection to use the same per-start masked subset as replay.
- Changed replay to use the same helper so the behavior is explicit and stays synchronized.
- Added a unit test that verifies the helper keeps only flagged reads and preserves input order.

The per-start cap itself did not change in this round. The change is strictly about making production and replay use the same subset-selection logic.

## Conclusion

Production and replay now agree on which reads are eligible for assembly under the current per-start mask.

This does not finish the caller parity work. The next active plan item should be allele likelihood marginalization, since it remains a confirmed output-facing gap and now has fewer surrounding filter inconsistencies.
