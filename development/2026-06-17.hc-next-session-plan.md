# Next-session plan: Rust HaplotypeCaller matching

## Unchanged aim

Build a Rust replacement for the module 6 GATK `HaplotypeCaller` stage for the current RNA-seq germline SNP/indel pipeline.

The Rust caller should produce output highly similar to Java GATK `HaplotypeCaller` on the same BAM, reference, dbSNP, and exon intervals, while remaining substantially faster with up to 40 threads and up to 128 GB RAM.

This is only for the current pipeline mode. Do not implement broad unused GATK features such as gVCF, joint genotyping, Spark, cloud I/O, or general GATK engine abstractions.

## Key conclusion

The remaining gap is not mostly indel representation or simple read clipping.

The current Rust caller is still essentially pileup-based. Java GATK `HaplotypeCaller` is not pileup-based; it is active-region assembly + haplotype generation + PairHMM + allele likelihood marginalization + genotyping.

Further threshold tuning or small evidence filters will not close the gap. The next useful work must implement a GATK-like algorithmic slice.

### Step 2: Implement true active-region grouping as a driver, not only replay metadata

Current Rust still scans each requested pileup locus. Next code should:

- compute per-locus active probability/ref-vs-any likelihood;
- smooth/group active loci into active regions;
- expand/pad active regions;
- collect finalized reads per active region;
- only generate candidates from active regions;
- preserve interval restrictions.

This alone may reduce some private calls but probably will not solve the main gap without assembly.

Do not add a dead `ActiveRegion` struct that is not wired into `call_variants`.

### Step 3: Implement minimal local haplotype generation for selected active regions

This is the highest-value next step.

Start with a constrained implementation for current RNA-seq regions:

- build a local reference haplotype for each active region;
- generate alternate haplotypes from observed SNP/short-indel events in the region;
- for simple regions, combine nearby events into haplotypes rather than calling each site independently;
- represent haplotypes with reference span, sequence, CIGAR/event map;
- emit replay haplotype tables comparable to GATK `gatk_debug.genotyper_haplotypes.tsv`.

A full `ReadThreadingAssembler` clone is large. A first constrained version can focus on:

- biallelic SNPs;
- short insertions/deletions already detected by pileup;
- nearby events within one active region;
- maximum haplotype count cap for speed.

The output should be better than independent pileup calls because nearby events and indels are represented on haplotypes.

### Step 4: Add a minimal PairHMM or alignment likelihood matrix

Without read-vs-haplotype likelihoods, genotype likelihoods and QUAL/PL/GQ will remain unlike GATK.

Implement a Rust likelihood engine that scores each finalized read against each candidate haplotype:

- start with a simple banded pair-HMM or pairwise alignment likelihood;
- use base qualities and indel qualities approximated from current data;
- cap base qualities by mapping quality where GATK does;
- emit `rust_replay_*.pairhmm.tsv` with rows comparable to GATK `pairhmm.tsv`.

Measure PairHMM table differences before changing final genotyping.

### Step 5: Implement allele likelihood marginalization and GATK-like genotype model

After haplotypes and read-haplotype likelihoods exist:

- map alleles/events to haplotypes;
- marginalize haplotype likelihoods to allele likelihoods per event;
- compute genotype likelihoods from allele likelihoods;
- apply SNP/indel heterozygosity priors similar to GATK;
- compute QUAL/GQ/PL in the same output model.

This should reduce shared-call genotype differences and QUAL/PL/QD divergence.

### Step 6: Revisit annotations only after likelihoods are closer

Current Rust annotations are simplified.

Only after likelihoods are closer, improve:

- `FS`
- `QD`
- `DP/AD`
- `DB`

Do not spend major time on annotations before assembly/PairHMM.
