## Round4 conclusion

- round4 confirmed that best-haplotype-pair genotyping helps the intended same-position indel-family problem:
  - `chr4:176328436`
  - `chr18:23482513`
  - `chr6:21596269`
  - `chr3:31532901`
  - `chr3:152463139`
  all matched Java much more closely in the full caller.
- But the global promotion was wrong for the whole callset:
  - Rust-only exploded from `3925` to `6457`.
  - Most of that blow-up was SNPs: `5283` Rust-only SNPs.
  - The round4 Rust-only sample was dominated by exact Java alleles that were present in Java with `PASS` but had `DP < 10`, which means the Rust global best-pair path was inflating event support into the good set rather than merely recovering truly missing calls.
- That makes round4 a useful diagnostic, not an acceptable stopping point.

## Round4 sample interpretation

- round4 Java-only sample:
  - `17/20` were `exact_absent`
  - `3/20` were `exact_present_nonpass`
- round4 Rust-only sample:
  - `16/20` were `exact_present_low_dp` in Java
  - `2/20` were `exact_present_nonpass`
  - `2/20` were `exact_absent`

This pattern is consistent with a region-global pair model being too aggressive for isolated/simple events.

## Round5 working hypothesis

- Keep pair-based assembled-event genotyping only where the evidence says it helps: overlapping or same-locus event families.
- For isolated assembled events, especially simple SNPs, use the per-event allele-marginalization path across all haplotypes instead of the region-global best pair.
- Acceptance target for round5:
  - keep the round4 improvements at the key indel-family loci;
  - reduce Rust-only sharply from the round4 `6457`;
  - specifically cut the Rust-only SNP inflation;
  - do not give back most of the Java-only gain.

## Round5 follow-up

- Focused preflight on exact original exon intervals showed that the hybrid boundary alone was not enough.
- The strongest example was `chr1:192716486 C>G`:
  - before read-span filtering, Rust mini full-caller still emitted `PASS 1/1 AD=0,23 DP=23`
  - Java at the same locus was `PASS 1/1 AD=0,5 DP=5`
- The code explanation was concrete: assembled-event genotyping still let every region read vote on every event, even when the read only supported some other nearby variant on the same haplotype and did not span the event itself.
- After requiring reads to overlap the event reference interval before contributing to assembled-event genotype likelihoods and AD/DP counts, the same exon-context preflight changed:
  - `chr1:192716486 C>G` -> Rust `PASS 1/1 AD=0,5 DP=5`, matching Java exactly
  - key indel-family loci stayed close to Java:
    - `chr4:176328436`
    - `chr18:23482513`
    - `chr6:21596269`
    - `chr3:31532901`
    - `chr3:152463139`
  - probe loci that were already below the good-set gate in Java stayed low, for example:
    - `chr2:203304551` Rust `DP=8`
    - `chr16:2232255` Rust `DP=5`
    - `chr18:13745168` Rust `DP=4`

This is the current best round6 candidate and it supersedes the earlier in-flight round5 tmux run, which was launched before the new read-span filter existed.

## Failed assembly experiment

- A focused assembly experiment was tried after round7 launch: do not use non-unique kmers as merge points in the Rust read-threading graph, and skip small-k reference graphs with repeated kmers.
- The motivation was real:
  - Java `ReadThreadingGraph` tracks `nonUniqueKmers`;
  - Java rejects low-complexity / cyclic k-mer attempts and retries larger `k`;
  - the remaining Rust replay miss at `chr7:23182192` still showed pathological repeat-ladder haplotypes.
- The narrow Rust implementation did not hold up on the six-locus Java-only follow-up replay:
  - known-good `pad100` replay baseline:
    - `active_regions=31`
    - `active_loci=47`
    - `candidate_events=36`
    - `genotype_rows=14`
  - experimental replay:
    - `active_regions=31`
    - `active_loci=47`
    - `candidate_events=12`
    - `genotype_rows=6`
- It did help one locus:
  - `chr3:190650707 C>CT` improved from replay event-only `0/0` to emitted genotype `PASS 0/1 DP=23 AD=12,11`
- But the aggregate regression was too large:
  - `chr7:23182192` still had `candidate_events=0`
  - `chr5:146128777` still had `candidate_events=0`
  - `chr22:43990401` still had `candidate_events=0`
  - total candidate events across the probe fell from `36` to `12`
- That experiment was reverted immediately.
- Current branch state is back to the verified `pad100` baseline, confirmed by replay re-run:
  - `active_regions=31`
  - `active_loci=47`
  - `candidate_events=36`
  - `genotype_rows=14`
- Conclusion:
  - the remaining assembly parity gap is real,
  - but a simple “non-unique kmer merge-point ban” port is not sufficient and is not safe to keep in the current Rust graph implementation.

## Zero-candidate fallback result

- After reverting the failed non-unique-kmer experiment, a narrower fallback was added:
  - only when assembly returns zero valid events;
  - collect strong pileup-derived candidate events from active loci;
  - require `depth >= 10` and at least `3` ALT-supporting observations;
  - build artificial local haplotypes from those fallback events and feed them through the normal PairHMM/genotyping path.
- Focused six-locus replay improved:
  - `candidate_events: 36 -> 38`
  - `genotype_rows: 14 -> 16`
- It recovered exact replay genotypes for:
  - `chr5:146128777 A>G`
  - `chr22:43990401 C>T`
- But that focused replay win did not yet translate into a broad full-callset rescue:
  - the representative six Java-only follow-up alleles are still exact-absent from round7 `rust.split.vcf.gz`;
  - round7 Java-only and Rust-only samples are still dominated by `exact_absent`.
- Conclusion:
  - keep the fallback because it is conservative and helps focused replay;
  - do not treat it as the main remaining fix for the full caller.
