# HaplotypeCaller Fetch-Window Padding Beyond Requested Intervals

## Summary

The Rust caller was building fetch windows directly from the requested `-L` intervals, then clipping active-region padding back to those same window bounds. That meant a locus near an exon or target boundary could never receive the caller's configured padding outside the requested interval, even though the caller thought it was adding `ACTIVE_REGION_PADDING`.

This is now fixed. Fetch windows are padded by the current `ACTIVE_REGION_PADDING` value before coalescing and partitioning, while the requested-interval gate still stays on the original intervals.

## Problem

Before this change:

- `coalesce_fetch_windows(...)` used the raw interval `start`/`end` as the fetch window.
- `pad_active_region(...)` clipped padded regions to `window.start` and `window.end`.
- For interval-edge loci, the effective padding could collapse to zero on one side.

That is especially risky for RNA-seq or exon-targeted interval lists, because assembly and haplotype extraction can need reference and read context just outside the requested interval.

## Evidence

### 1. Java-only sample was enriched at interval edges

From the first `20` sampled Java-only good variants:

- `19` were Rust `exact_absent`.
- `11/19` exact-absent loci were within `50 bp` of an original interval edge.
- `12/19` exact-absent loci were within `100 bp` of an original interval edge.

This did not prove the root cause by itself, but it made interval-boundary context a high-value target.

### 2. Centered replay recovered context that original exon replay lacked

For `chr18:23482513 GA>G`:

- Original exon replay before the fix had active region row:
  - padded region `chr18:23482463-23482528`
  - `candidate_events=0`
- Centered replay on a custom `pos +/- 150` interval had active region row:
  - padded region `chr18:23482463-23482578`
  - `candidate_events=15`

That difference was consistent with the exon-boundary clip suppressing local assembly context.

### 3. Original exon replay improved after the code change

Using a 3-locus original-exon replay probe:

- before change: `candidate_events=27`
- after change: `candidate_events=47`

For the same `chr18:23482513 GA>G` locus:

- before: exact replay event absent
- after: exact replay event present

The final genotype was still not emitted for that locus, so this change fixes one boundary-related candidate-generation problem, not the whole Java/Rust parity gap.

### 4. Full good-set comparison improved slightly but not enough

The first full module 6 rerun after this change was recorded in:

- `/XCLabServer002_fastIO/gatk-faster-rnaseq/SRR949115_broad_hg38_run2/module6_good_compare_20260617_round2`

Compared with the corrected round1 baseline:

| metric | round1 | round2 |
|---|---:|---:|
| Java-good | `12327` | `12327` |
| Rust-good | `14229` | `14331` |
| shared | `10950` | `10977` |
| Java-only good | `1377` | `1350` |
| Rust-only good | `3279` | `3354` |
| Java sensitivity | `88.829%` | `89.048%` |
| Rust precision vs Java | `76.956%` | `76.596%` |
| GT diff | `120` | `119` |

Interpretation:

- the boundary fix recovered some real Java-good calls;
- it slightly improved shared exact allele keys and slightly reduced Java-only good variants;
- it also increased Rust-only good variants, so it is not sufficient as a standalone parity fix.

## Code Change

In [src/rust/src/haplotype_caller.rs](/data/p/gatk/gatk-faster-rnaseq-germline-snps-indels/src/rust/src/haplotype_caller.rs):

- added `padded_fetch_bounds(interval, dict)`
- changed `coalesce_fetch_windows(...)` to use padded bounds
- kept `window.intervals` unchanged so `position_is_requested(...)` still restricts active-locus discovery to the original requested targets

This keeps the behavioral change narrow:

- more context for padding and assembly near target edges
- no expansion of the requested calling set itself

Result:

Focused replay probe:

- before-change original-exon replay: reproduced missing `chr18:23482513 GA>G` candidate event
- after-change original-exon replay: recovered the exact replay event at that locus

## Remaining Gap

The centered Java-only replay audit still showed:

- `9/20` sampled loci recovered as exact candidate events
- only `1/20` recovered as exact final genotypes

So the next work should stay evidence-driven and focus on the dominant remaining post-assembly differences, not assume that interval padding alone is enough.
