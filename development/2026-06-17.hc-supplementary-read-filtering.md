# HaplotypeCaller supplementary-read filtering parity fix

## Aim

Bring the Rust HaplotypeCaller read filter closer to Java GATK for RNA-seq input after `SplitNCigarReads`.

The concrete question for this round was whether the Rust full caller was incorrectly discarding supplementary alignments that Java HaplotypeCaller would still use.

## Source check

Local source verification used:

- Rust caller read collection in `src/rust/src/haplotype_caller.rs`
- Rust prefilter in `src/rust/src/bin/rust_hc_prefilter.rs`
- Java HaplotypeCaller defaults in `/data/p/gatk/gatk/src/main/java/org/broadinstitute/hellbender/tools/walkers/haplotypecaller/HaplotypeCallerEngine.java`
- Java `SplitNCigarReads` in `/data/p/gatk/gatk/src/main/java/org/broadinstitute/hellbender/tools/walkers/rnaseq/SplitNCigarReads.java`
- Java SA-tag repair in `/data/p/gatk/gatk/src/main/java/org/broadinstitute/hellbender/utils/SATagBuilder.java`

Confirmed findings:

- Rust `scan_call_window` and `scan_replay_window` were both skipping `record.is_supplementary()`.
- Rust `read_passes_hc_filter()` already had the better design: supplementary reads are included unless explicitly excluded.
- Rust `rust_hc_prefilter` already defaults to including supplementary reads and only excludes them with `--exclude-supplementary`.
- Java HaplotypeCaller standard filters include mapped, MQ, not-secondary, not-duplicate, vendor-QC, non-zero-reference-length, good-CIGAR, and wellformed filters, but not `NOT_SUPPLEMENTARY_ALIGNMENT`.
- Java `SplitNCigarReads` keeps one split segment primary and marks the non-primary split segments supplementary via `SATagBuilder.setReadsAsSupplemental(...)`.

Conclusion from source: the Rust full caller was too strict for RNA-seq split-read input.

## Code changes

Changed files:

- `src/rust/src/haplotype_caller.rs`
- `src/rust/src/bin/rust_haplotype_caller.rs`
- `src/rust/src/bin/rust_hc_region_replay.rs`
- `src/step_haplotype_caller.py`

Implementation:

- Added `exclude_supplementary: bool` to `HaplotypeCallerConfig`.
- Added `--exclude-supplementary` to:
  - `rust_haplotype_caller`
  - `rust_hc_region_replay`
  - `src/step_haplotype_caller.py` for the Rust backend
- Switched production read collection and replay read collection to use `read_passes_hc_filter(...)` instead of manually rejecting supplementary reads.
- Threaded `exclude_supplementary` through active-region discovery and pileup evidence helpers so debug behavior is internally consistent.
- Added unit tests that verify:
  - supplementary reads pass by default;
  - supplementary reads are excluded only when explicitly requested;
  - reads without reference-consuming CIGAR ops are still rejected.

## Conclusion

This round fixes one confirmed Java/Rust filter mismatch that is specifically risky for RNA-seq post-`SplitNCigarReads` input.

The parity objective is still incomplete. The next high-value items remain:

- production/replay read subset consistency;
- allele likelihood marginalization;
- replay genotype emission;
- then another full Java-vs-Rust good-variant comparison round.
