# Java-only Replay Genotyping Analysis After Fetch-Window Padding Fix

## Summary

The fetch-window padding fix recovered a real class of interval-boundary candidate misses, but it did not resolve the dominant post-assembly mismatch in the sampled Java-only loci.

For the first `20` sampled Java-only good variants:

- centered replay recovered the exact Rust candidate event for `9/20`;
- centered replay recovered the exact final Rust genotype at the default calling threshold for only `1/20`;
- zero-threshold replay (`--standard-min-confidence-threshold-for-calling 0`) showed that `8/9` recovered exact events were still present in the final call table but were genotyped as `0/0` with `QUAL=0`.

That means the next dominant problem is not just final filtering. It is event genotyping in complex local assemblies.

## Key Finding

The relevant failure mode is:

- exact Java-good allele is absent from Rust raw VCF in the full run;
- replay on a centered local interval recovers the exact Rust candidate event;
- replay with zero calling threshold still emits that exact allele as `0/0`, not as a low-quality non-reference call.

This narrows the next work to the assembled-event genotyping model.

### Candidate-event clustering

Several of the recovered exact-event loci sit inside large same-position or near-position candidate clusters in Rust replay:

| locus | replay candidate pattern |
|---|---|
| `chr18:23482513` | `15` same-position indel candidates |
| `chr6:21596269` | `12` same-position indel candidates |
| `chr3:31532901` | `6` same-position indel candidates |
| `chr3:152463139` | `15` same-position indel candidates |
| `chr19:18280054` | `53` candidate events within +/-20 bp |

This is consistent with the current Rust design:

- events are genotyped mostly one-by-one as independent biallelic variants;
- haplotypes not containing the current event are treated as REF for that event;
- competing alternative haplotypes in repetitive regions can therefore act like reference evidence against the exact Java-good allele.

## Interpretation

The current evidence supports the plan item on multi-allelic and overlapping-event genotyping more strongly than any remaining padding-only explanation.

The next change should target one of these:

1. group same-locus / overlapping events before genotyping;
2. jointly genotype competing alleles at that locus;
3. if a full joint model is still too large, stop counting competing same-locus/overlapping alternative haplotypes as pure REF evidence for each individual event.

## Conclusion

The fetch-window padding fix should still be kept; it solved a real candidate-generation boundary defect.

But the next parity fix should target complex assembled-event genotyping, not the confidence threshold alone.
