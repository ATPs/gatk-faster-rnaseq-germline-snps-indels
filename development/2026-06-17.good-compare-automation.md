# Module 6 good-compare automation helpers

## Aim

Make the current Java-vs-Rust good-set comparison round easier to rerun and easier to continue to the required 20/20 sampling step without retyping ad hoc shell commands.

## Changes

Added helper scripts in `development/`:

- `2026-06-17.module6-good-compare-round1.zsh`
  - runs the Rust module 6 caller with the current release binary;
  - applies the Java hard filters;
  - splits multi-ALT records;
  - builds the `PASS && DP>=10 && AC>=3` good sets;
  - runs `rust_hc_vcf_compare`.
- `2026-06-17.good-compare-postprocess.zsh`
  - reads `java_vs_rust.good.summary.tsv`;
  - prints the `pass` summary row;
  - generates:
    - `round1.rust_only.sample20.tsv`
    - `round1.java_only.sample20.tsv`

For the in-flight round started before the script update, a second detached `tmux` watcher session was launched to run the post-process helper as soon as `java_vs_rust.good.summary.tsv` appears.

## Verification

Checks run:

```bash
zsh -n development/2026-06-17.module6-good-compare-round1.zsh
zsh -n development/2026-06-17.good-compare-postprocess.zsh
tmux has-session -t hc_good_compare_20260617_r1_162534
tmux has-session -t hc_good_post_20260617_r1_163540
```

Results:

- both scripts passed shell syntax checks;
- the main comparison session is running;
- the post-process watcher session is running.

## Conclusion

This does not change Rust caller behavior. It reduces operator friction around the exact workflow required by the plan file and makes the current round easier to finish and inspect once the long run completes.
