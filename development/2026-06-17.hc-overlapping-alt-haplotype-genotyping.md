# HaplotypeCaller Overlapping-Alt Haplotype Genotyping Approximation

## Summary

This change addresses one specific bias in Rust assembled-event genotyping:

- before this change, every haplotype that did not contain the current event was counted as REF evidence for that event;
- that included haplotypes carrying competing overlapping alternative events;
- in repetitive indel clusters, those competing alternative haplotypes could force an exact recovered Java-good allele back to `0/0`.

The current fix is narrow:

- when genotyping one event, haplotypes carrying overlapping competing events are excluded from the REF bucket for that event.

This is not full multi-allelic joint genotyping, but it is a better approximation than treating overlapping ALT haplotypes as pure reference evidence.

## Problem

From centered replay on the first `20` sampled Java-only good variants after the fetch-window padding fix:

- exact replay candidate event recovered for `9/20`;
- exact final non-reference replay genotype recovered for only `1/20`;
- zero-threshold replay showed that `8/9` recovered exact events were still present in the final genotype table but as `0/0`, not just filtered out.

The strongest examples were repetitive indel clusters:

- `chr18:23482513` had `15` same-position indel candidates;
- `chr6:21596269` had `12`;
- `chr3:31532901` had `6`;
- `chr3:152463139` had `15`;
- `chr19:18280054` sat inside a `53`-event +/-20 bp cluster.

In that situation, the old logic was structurally biased:

- current event ALT likelihood used only haplotypes containing that event;
- current event REF likelihood used every other haplotype;
- overlapping alternative haplotypes were therefore treated like REF evidence against the current event.

## Code Change

In [src/rust/src/haplotype_caller.rs](/data/p/gatk/gatk-faster-rnaseq-germline-snps-indels/src/rust/src/haplotype_caller.rs):

- added `events_overlap(...)`;
- updated `genotype_assembled_events(...)` so overlapping competing-event haplotypes are excluded from the REF likelihood bucket for the current event.

This keeps the change small and local to assembled-event genotyping.

## Focused Verification

### Unit test

Added a focused unit test where:

- one haplotype supports ALT1;
- another haplotype supports a competing overlapping ALT2;
- old logic would let ALT2 act like REF evidence against ALT1;

### Centered zero-threshold replay

Reran:

```bash
src/rust/target/debug/rust_hc_region_replay \
  -I /XCLabServer002_fastIO/gatk-faster-rnaseq/SRR949115_broad_hg38_run2/baseline/SRR949115.recal.bam \
  -R /data1/pub/gatk/broad_hg38/Homo_sapiens_assembly38.fasta \
  -L /XCLabServer002_fastIO/gatk-faster-rnaseq/SRR949115_broad_hg38_run2/module6_good_compare_20260617_round1/java_only20_replay_20260617/java_only20.regions.interval_list \
  -O /tmp/java_only20_replay_overlapfix_minconf0/replay \
  --dbsnp /data1/pub/gatk/broad_hg38/Homo_sapiens_assembly38.dbsnp138.vcf.gz \
  --dont-use-soft-clipped-bases \
  --standard-min-confidence-threshold-for-calling 0 \
  --threads 8 \
  --memory-gb 32 \
  --native-pair-hmm-threads 4
```

Exact final non-reference recovery among the sampled Java-only loci improved:

- before change: `1/20`
- after change: `3/20`

Concrete improvements:

| locus | before | after |
|---|---|---|
| `chr19:18280054 A>G` | `0/0`, `QUAL=0`, `AD=799,95` | `0/1`, `QUAL=613`, `AD=775,107` |
| `chr6:21596269 C>CA` | `0/0`, `QUAL=0`, `AD=133,15` | `0/1`, `QUAL=20`, `AD=67,18` |

Unchanged recovered-but-`0/0` examples:

- `chr18:23482513 GA>G`
- `chr3:31532901 T>TTCCTCC`
- `chr3:152463139 CT>C`
- `chr5:37605101 C>CA`

So the change is useful but incomplete.
