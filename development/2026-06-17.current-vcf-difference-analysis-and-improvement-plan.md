## Code Findings

Current Rust code has substantial new assembly and PairHMM pieces, but important GATK behaviors remain simplified:

- `src/rust/src/haplotype_caller.rs` emits one biallelic `VariantCall` at a time; there is no merged per-locus multi-allelic record or PL vector for more than one ALT allele.
- `scan_call_window` maps haplotypes to a candidate event and uses the best haplotype containing or not containing that event as ALT/REF likelihoods. GATK instead builds an allele mapper, marginalizes read-haplotype likelihoods to all alleles at a merged event, and genotypes the complete allele set.
- `scan_call_window` caps assembly input with `read_bases_list.iter().take(500)`. This is a valid runtime guard in spirit, but it is not the same as Java GATK's default read limiting, which uses positional downsampling with `--max-reads-per-alignment-start 50` before reads are assembled.
- PairHMM currently uses fixed insertion/deletion qualities (`45`) and constant GCP (`10`) rather than GATK's full transformed read qualities, PCR error model, dynamic read disqualification, and read-quality capping path.
- `assemble_haplotypes` uses the same default requested k-mer sizes as Java GATK (`10,25`) and stops early once a non-reference graph is found. That early-stop idea is also present in Java when the graph is adequate. The remaining difference is not the existence of the speed shortcut; it is that Java can expand k-mers on cycles/failed/suspect graphs, uses pruning and dangling-end recovery, and defaults to a much larger haplotype cap (`maxNumHaplotypesInPopulation=128`) than Rust's current graph path cap of 10.
- `write_variant_record` always writes biallelic `PASS` raw records; filter status only appears after downstream `VariantFiltration`.

## Follow-up Java Assembler Check

The earlier wording around Rust assembly limits needed nuance. Java GATK uses several similar runtime-bounding ideas:

- Default assembler k-mer sizes are `10,25` in `ReadThreadingAssemblerArgumentCollection`.
- Default `--max-reads-per-alignment-start` is `50` in `AssemblyRegionArgumentCollection`, applied by `AssemblyRegionWalker.createDownsampler()` as a positional downsampler.
- Default `maxNumHaplotypesInPopulation` is `128`.
- `ReadThreadingAssembler` expands k-mers by 10 for up to 6 more attempts when cycles or failed graphs require it, but it can stop once it has an adequate graph or only reference haplotype.
- Java prunes low-weight chains and attempts dangling-head/tail recovery before haplotype finding.

Therefore the Rust `take(500)` and `[10,25]` choices should not be treated as inherently wrong. The improvement target is to make the shortcuts Java-like and measurable: use positional/read-start downsampling instead of input-order truncation, keep k-mer expansion/fallback only for suspect or failed graphs, and expose haplotype/path caps so selected-region replay can show whether they are causing missed events.

Direct verdict for the current Rust settings:

- Using requested k-mer sizes `10,25` is aligned with Java GATK defaults.
- Stopping early for an adequate graph is aligned with Java GATK's bounded assembly strategy.
- Keeping an assembly/read cap for speed is necessary and aligned with Java's design goal of avoiding excessive PairHMM work.
- The Rust-specific part is `take(500)` by active-region input order. Java instead uses deterministic positional downsampling: by default, at most 50 reads per alignment start before assembly-region processing.
- The Rust-specific risk is stopping after any non-reference graph rather than after a graph is adequate and not suspect. Java can expand to larger k-mers when cycles, failed graphs, low-complexity graphs, or suspect haplotypes indicate the first graph is not reliable.
- The plan should therefore keep the runtime-saving limits, but change them to Java-like positional downsampling, Java-like graph adequacy checks, and measured path/haplotype caps.

## Java HaplotypeCaller Work Path to Match

The Rust plan should align to the Java control flow below, not to a generic local-assembly caller.

### 1. Assembly-region traversal and read limiting

Java `AssemblyRegionWalker` applies the normal read filters and, by default, creates a `PositionalDownsampler` from `AssemblyRegionArgumentCollection.maxReadsPerAlignmentStart`.

Relevant defaults:

- `minAssemblyRegionSize = 50`
- `maxAssemblyRegionSize = 300`
- `assemblyRegionPadding = 100`
- `maxReadsPerAlignmentStart = 50`
- active probability threshold `0.002`
- probability propagation distance `50`

This means Java limits high-depth evidence by alignment start, not by taking the first N reads in the active region. That distinction matters for RNA-seq pileups because input-order truncation can be biased by BAM sort order and mate/read ordering.

### 2. Active-region scoring

`HaplotypeCallerEngine.isActive()` computes a ref-vs-any likelihood at each locus with `ReferenceConfidenceModel.calcGenotypeLikelihoodsOfRefVsAny()`, then converts that through `MinimalGenotypingEngine` into an activity probability. These activity states are smoothed and grouped by the engine into bounded active regions.

Rust does not need a full GATK engine clone, but the active-region replay should report the comparable pieces: locus activity score, final active-region bounds, padded assembly span, and read counts after downsampling/finalization.

### 3. Read finalization before assembly

`AssemblyBasedCallerUtils.assembleReads()` first calls `finalizeRegion()`.

That method:

- hard-clips soft clips when `dontUseSoftClippedBases` requires it, otherwise may restore soft-clipped bases when fragment information is well-defined;
- clips low-quality tails using the caller's minimum tail quality;
- clips adapters;
- hard-clips reads to the padded assembly region;
- removes empty/non-overlapping reads;
- sorts finalized reads by coordinate;
- corrects overlapping mates with `cleanOverlappingReadPairs()`;
- tracks a hard-clipped pileup-read list for pileup/event logic.

This is a major evidence-definition step. Rust read handling should be compared here before tuning variant thresholds.

### 4. Local assembly

Java `ReadThreadingAssembler.runLocalAssembly()` builds a reference haplotype over the padded reference interval, optionally error-corrects reads, then uses `ReadThreadingAssembler`.

Important default controls:

- requested k-mer sizes are `10,25`;
- graph cycle or failed/suspect graph recovery can expand k-mer size by `10` for up to `6` additional attempts;
- `minPruneFactor = 2`;
- dangling tails and heads are recovered by default;
- `minDanglingBranchLength = 4`;
- `maxNumHaplotypesInPopulation = 128`;
- reference haplotype is preserved in the result set.

Java may stop once it has an adequate graph or only the reference haplotype. It does not expand k-mers just for completeness. The Rust equivalent should preserve this bounded behavior while adding Java-like failure/suspect-graph fallback.

### 5. Trimming and read-haplotype likelihoods

In `HaplotypeCallerEngine.callRegion()`, Java:

- collects assembled events, pileup events, and given alleles;
- trims the assembly result to the variant-containing region;
- removes reads shorter than `MINIMUM_READ_LENGTH_AFTER_TRIMMING`;
- filters reads for genotyping;
- computes read-vs-haplotype likelihoods across all retained haplotypes using the configured likelihood engine, normally PairHMM;
- optionally filters weak alleles/haplotypes;
- realigns reads to their best haplotype for downstream annotation/BAM output.

Rust currently computes PairHMM-like scores, but its inputs and downstream use are simplified. The next Rust work should keep the complete read-haplotype matrix through genotyping instead of reducing too early to one event's best ref/alt haplotype.

### 6. Per-locus genotyping

`HaplotypeCallerGenotypingEngine.assignGenotypeLikelihoods()` is the key Java path for matching output VCFs:

- `EventMap.buildEventMapsForHaplotypes()` maps haplotype differences to variant events.
- For each event start, Java gets events from active haplotypes.
- `AssemblyBasedCallerUtils.makeMergedVariantContext()` merges alleles at the locus into one variant context.
- `AssemblyBasedCallerUtils.createAlleleMapper()` maps each allele to all haplotypes supporting it.
- `readLikelihoods.marginalize(alleleMapper)` converts read-haplotype likelihoods into read-allele likelihoods.
- Evidence is retained only if it overlaps the variant-calling relevant interval.
- `calculateGLsForThisEvent()` computes genotype likelihoods for the complete allele set.
- `calculateGenotypes()` chooses the call.
- `makeAnnotatedCall()` adds annotations using the read-allele likelihoods and pre-filtered likelihoods.

This is where Rust currently differs most: it emits one biallelic event at a time and uses a best-haplotype shortcut instead of merged multi-allelic allele likelihood marginalization.

## Improvement Plan

### 1. Lock Java-aligned diagnostics before more algorithm changes

Add or preserve a split-ALT feature comparison command so every HaplotypeCaller change reports:

- record-level `rust_hc_vcf_compare`;
- split-ALT exact-key counts;
- private PASS type/genotype/dbSNP/depth/cluster summaries;
- exact-key PASS/non-PASS filter disagreement summaries;
- shared GT-difference matrix.

Also extend selected-region replay so it reports Java checkpoints:

- active-region score and final bounded active-region span;
- padded assembly span;
- input reads, downsampled reads, finalized reads, short-read removals, and genotyping reads;
- haplotypes with k-mer size and graph status;
- read-haplotype matrix;
- allele mapper and read-allele matrix per event.

Success check: reproduce this note's current split-ALT baseline and selected-region stage counts before changing caller behavior.

### 2. Match Java read limiting and `finalizeRegion()`

Target Java:

- `AssemblyRegionWalker.createDownsampler()`
- `AssemblyBasedCallerUtils.finalizeRegion()`
- `AssemblyBasedCallerUtils.cleanOverlappingReadPairs()`

Concrete work:

- replace the unconditional first-500-read assembly cap with deterministic positional downsampling, starting from Java's default `--max-reads-per-alignment-start 50`;
- keep a bounded assembly input path for speed, but report retained reads per active region and per alignment start;
- hard-clip or restore soft clips according to the Java conditions;
- apply low-quality tail clipping, adapter clipping, and hard clipping to the padded assembly region;
- remove empty/non-overlapping reads and reads below Java's minimum trimmed length;
- add overlapping mate base-quality correction.

Why: Java also limits work, but it limits reads by alignment start after normal read filtering. Matching this should reduce order-dependent evidence differences while preserving the runtime guard the Rust caller needs.

Success check: selected-region read/finalization counts move closer to Java debug output, and Rust-private low-depth/no-db clustered calls do not increase.

### 3. Match Java assembly control while preserving bounded runtime

Target Java:

- `ReadThreadingAssemblerArgumentCollection`
- `ReadThreadingAssembler.runLocalAssembly()`
- `ReadThreadingAssembler.assembleGraphsAndExpandKmersGivenHaplotypes()`
- graph pruning and dangling-end recovery before haplotype finding

Concrete work:

- keep Java's default requested k-mer sizes `10,25`;
- keep early stop for an adequate graph;
- add Java-like k-mer expansion only for cycles, failed graphs, low-complexity graphs, or suspect haplotypes;
- add low-weight-chain pruning and dangling-head/tail recovery;
- compare Rust path cap 10 with Java default `maxNumHaplotypesInPopulation=128`;
- if 128 is too expensive, measure a sensitivity/runtime curve on selected regions and choose an explicit cap with evidence.

Why: the Rust speed shortcut is acceptable, but it should fail over like Java when the first graph is incomplete or suspect.

Success check: selected-region GATK-private event count decreases, especially no-nearby SNPs and short indels, without a large Rust-private cluster expansion.

### 4. Preserve Java-style read-haplotype likelihoods through genotyping

Target Java:

- `HaplotypeCallerEngine.callRegion()`
- `PairHMMLikelihoodCalculationEngine.computeReadLikelihoods()`
- optional allele filtering before genotyping

Concrete work:

- feed PairHMM with Java-like read bases, base qualities, insertion/deletion qualities, GCP, mapping-quality caps, and dynamic read-disqualification behavior;
- keep the full read-haplotype likelihood matrix for all retained haplotypes;
- avoid converting directly from haplotype likelihoods to a single event-level best-ref/best-alt pair;
- emit replay TSVs for unnormalized and normalized read-haplotype likelihoods.

Why: Java's later allele marginalization depends on the full haplotype matrix. Early collapse loses information, especially at multi-allelic and linked-event sites.

Success check: shared-call QUAL/GQ/QD deltas shrink and high-outlier DP/AD cases become easier to explain from replay.

### 5. Implement Java-like per-locus allele merging and multi-allelic genotyping

Target the current biallelic/event-by-event caller first.

Implement a minimal Rust analogue of:

- `AssemblyBasedCallerUtils.makeMergedVariantContext()`
- `AssemblyBasedCallerUtils.createAlleleMapper()`
- `AlleleLikelihoods.marginalize(alleleMapper)`
- `HaplotypeCallerGenotypingEngine.calculateGLsForThisEvent()`

Concrete work:

- group candidate events by locus;
- emit one VCF record with all compatible ALT alleles at that locus;
- compute diploid PLs over all genotype combinations for N alleles;
- map alleles to all haplotypes supporting them and marginalize read-haplotype likelihoods into read-allele likelihoods;
- retain only evidence overlapping Java's variant-calling relevant interval;
- assign `GT`, `AD`, `DP`, `GQ`, `AC`, `AF`, `AN`, `QD`, and `FS` from the multi-allelic likelihood model;
- preserve biallelic output only when there is one real ALT allele.

Why this matters: it directly addresses 70 multi-ALT GATK-private allele keys and the 982 shared GT differences, especially the 774 `1/1 -> 0/1` undercalls.

Selected-region success check: genotype-diff count decreases without increasing Rust-private events.

Full-run success check: PASS sensitivity improves and precision does not regress materially.

### 6. Annotation/filter parity as a final correctness pass

Do this after improving evidence and genotype likelihoods.

Concrete work:

- verify `QD` uses the same QUAL/depth definition as GATK for this pipeline;
- compare `FS` and add `SOR` if needed for debugging, even if not used by current hard filters;
- add `MQ`, `BaseQRankSum`, `MQRankSum`, and `ReadPosRankSum` only if later filters or parity reports need them;
- confirm `VariantFiltration --window 35 --cluster 3` sees equivalent raw VCF structure after multi-allelic merging.

Why last: the biggest discrepancies are candidate/genotyping/evidence, and threshold tuning would hide those problems.

## Detailed Code Audit (2026-06-17)

This section records concrete findings from reading the Rust source files line-by-line and comparing them against the Java control flow described above. File paths and line numbers are as of the current HEAD.

### `src/rust/src/haplotype_caller.rs`

#### Active-region padding is 50 instead of Java's 100

`ACTIVE_REGION_PADDING` at line 33 is `50`. Java's `AssemblyRegionArgumentCollection.DEFAULT_ASSEMBLY_REGION_PADDING` is `100` (verified: `AssemblyRegionArgumentCollection.java:24`). HaplotypeCaller does not override this value (`HaplotypeCallerArgumentCollection.java` has no `defaultAssemblyRegionPadding` override). This means every active region handed to the Rust assembler is 50 bp shorter on each side than Java's padded assembly span. Short-read evidence that falls inside the Java padding window but outside the Rust padding is never assembled. This is a leading candidate cause of the 4,587 GATK-private PASS allele keys that have no Rust PASS call within 100 bp.

Fix: raise `ACTIVE_REGION_PADDING` to `100` and add a replay column for padded span so the change is measurable.

#### Soft clips silently dropped when building assembly reads

In `scan_call_window` lines 933–958, reads are decomposed into bases for assembly:

```rust
SoftClip(_) => {
    seq_idx += len;
}
```

Soft-clipped bases are always skipped regardless of `config.dont_use_soft_clipped_bases`. GATK's `finalizeRegion()` restores soft-clipped bases when `dontUseSoftClippedBases` is false and fragment information is consistent. In RNA-seq data this matters because junction-spanning reads frequently have soft clips at intron boundaries. Those bases carry real variant signal that never enters the de Bruijn graph.

Fix: when `dont_use_soft_clipped_bases` is false, include soft-clipped bases in the assembly read sequence, matching the existing `clip_read_for_evidence` logic that already handles this correctly for pileup evidence.

#### Quality floor of 18 is not a GATK parameter

Lines 941–942 apply `.min(mapq).max(18)` to every base quality fed to PairHMM. The `.min(mapq)` cap is correct (Java also caps base quality at mapq). The `.max(18)` floor is not a standard GATK parameter. Java imposes no such minimum; it trusts base qualities as reported after BQSR. The floor can inflate low-quality evidence and may be contributing to Rust-private low-depth calls.

Fix: remove `.max(18)` or replace with a documented and configurable minimum base quality (Java default effective minimum is the `minBaseQualityScore` filter at 10, already enforced by `PIPELINE_MIN_BASEQ`).

#### Assembly read cap uses input order, not positional downsampling

Line 972:

```rust
let assembly_reads: Vec<&Vec<u8>> = read_bases_list.iter().take(500).collect();
```

Reads are filtered by input order. In a coordinate-sorted BAM with high-depth RNA-seq pileups, reads from the first alignment starts are overrepresented. Java's `PositionalDownsampler` caps at `maxReadsPerAlignmentStart=50` after normal read filters, which is uniform across positions. The PairHMM scoring loop (lines 988–1010) runs on the full `read_bases_list` without the 500-read cap, so PairHMM and assembly see different read sets. This is a logic inconsistency on top of the order-bias problem.

Fix: implement positional downsampling keyed on `record.pos()` with a default cap of 50 reads per alignment start, applied to the same list used for both assembly and PairHMM. Keep a tunable constant `MAX_READS_PER_ALIGNMENT_START`.

#### PairHMM insertion/deletion quality fallback is correct, but BI/BD BAM tags are not read

Lines 993–995:

```rust
let read_ins_quals = vec![45; r_bases.len()];
let read_del_quals = vec![45; r_bases.len()];
let gcp = 10;
```

The default values `45` and `10` are correct. Java's `ReadUtils.DEFAULT_INSERTION_DELETION_QUAL = 45` (verified: `ReadUtils.java:44`) and `LikelihoodEngineArgumentCollection.gcpHMM = 10` (verified: `LikelihoodEngineArgumentCollection.java:63`) use the same values as fallbacks. Java's `getBaseInsertionQualities` and `getBaseDeletionQualities` first check for BQSR-written BAM tags `BI` and `BD` (verified: `ReadUtils.java:822,829`) and fill with `45` only when those tags are absent. For post-BQSR BAMs (which this pipeline uses), the `BI`/`BD` tags carry per-base indel qualities that vary by read. Using a flat `45` for all reads ignores this per-read signal and makes every read's indel transition probability identical.

Fix: read the `BI` and `BD` BAM auxiliary tags when present (they are stored as ASCII-encoded quality strings, same format as the `BQ` tag) and fall back to `45` only when absent. GCP (`10`) needs no change — Java uses the same constant value.

The `gcp` value is passed as a scalar in Rust but Java applies it per-base via `StandardPairHMMInputScoreImputator.newInstance(constantGCP)` which fills a per-base array of the same constant — functionally equivalent to the current Rust approach. No change needed for GCP.

#### Genotyping uses best-haplotype shortcut instead of allele marginalization

Lines 1027–1039 implement genotyping for each event:

```rust
for hap_idx in 0..local_haplotypes.len() {
    let lhood = read_haplotype_likelihoods[read_idx][hap_idx];
    if hap_contains_allele[hap_idx] {
        if lhood > best_alt_lhood { best_alt_lhood = lhood; }
    } else {
        if lhood > best_ref_lhood { best_ref_lhood = lhood; }
    }
}
read_allele_likelihoods_ref[read_idx] = best_ref_lhood;
read_allele_likelihoods_alt[read_idx] = best_alt_lhood;
```

This takes the single best haplotype likelihood for each allele class (ALT-containing or REF) and treats that as the read-allele likelihood. Java's `AlleleLikelihoods.marginalize(alleleMapper)` sums over all haplotypes in each allele class in probability space (not max in log space). Max-in-log-space underestimates allele likelihoods when multiple haplotypes support the same allele and overestimates the distinction between alleles when the best ALT haplotype also happens to carry other events that confuse the REF haplotypes. This is the primary mechanism behind the 774 `1/1→0/1` shared genotype undercalls: when a homozygous-alt site has multiple ALT haplotypes (e.g. phased with a nearby SNP), the `best_ref_lhood` picks the best REF haplotype and the signal for homozygosity is diluted.

Fix: replace max-in-log-space with log-sum-exp marginalization over all haplotypes in each allele class. This is the single highest-impact genotyping fix and requires no new data structures beyond what already exists.

#### Likelihood accumulation over reads mixes log10 and natural scales incorrectly for the het case

Line 1067:

```rust
log10_likelihoods[1] += log10_sum_exp(&[rL, aL]) - 0.3010299956639812;
```

`0.3010299956639812` is `log10(2)`. This correctly models the het likelihood as `0.5 * P(read|REF) + 0.5 * P(read|ALT)` in log10 space. However, `rL` and `aL` are themselves the raw log10 PairHMM scores for the best-haplotype shortcut described above, not the marginalized allele likelihoods. The formula is correct in structure but operates on inputs that already carry the max-in-log-space distortion. Fixing the allele marginalization (above) fixes this too.

#### Replay window uses pileup-based events, not assembly events

`scan_replay_window` (lines 1123–1416) generates candidate events via `best_snp_call` and `best_indel_call` (pileup-based genotyping, lines 1279–1301), whereas `scan_call_window` uses `assemble_haplotypes`. The replay tables therefore represent a different calling path than the production output. Haplotype and PairHMM rows are computed in the replay path from a `build_local_haplotypes` call (lines 1308–1315) that constructs haplotypes from pileup events rather than from graph assembly.

Fix: the replay path should call `assemble_haplotypes` and use the resulting haplotypes and events, matching the production path. The current divergence means replay comparisons are not measuring the same algorithm that produces the VCF.

### `src/rust/src/assembly.rs`

#### Haplotype path cap is 10, Java default is 128

`assemble_haplotypes` (haplotype_caller.rs line 3744) calls:

```rust
let best_paths = graph.find_best_haplotypes(source_idx, sink_idx, 10);
```

Java's `maxNumHaplotypesInPopulation` defaults to `128` (verified: `ReadThreadingAssemblerArgumentCollection.java:112`). In active regions with multiple nearby variants (common in RNA-seq exons), the Rust assembler discards haplotypes that carry GATK-private events. Raising this cap to 128 should increase sensitivity, especially for indels and for linked SNPs where the correct haplotype is not among the top 10 by graph score.

Fix: raise cap to 128 and expose as a configurable constant. If runtime increases materially, profile on selected GATK-private regions and pick an evidence-based intermediate (e.g. 32 or 64).

#### No graph pruning and no dangling-end recovery

`ReadThreadingGraph.add_sequence` and `find_best_haplotypes` in `assembly.rs` have no edge-weight pruning and no dangling head/tail recovery. Java's `ReadThreadingAssembler` applies (all verified in `ReadThreadingAssembler.java:688-689`, `ReadThreadingAssemblerArgumentCollection.java:92,125`):

- `minPruneFactor = 2`: edges with multiplicity below this are pruned before haplotype finding. Removes low-support noise paths that inflate graph complexity.
- Dangling tail recovery (`recoverDanglingTails`): short branches that don't reach the sink kmer are extended with reference bases when a path can be found. This recovers indel haplotypes that are cut short by read ends.
- Dangling head recovery (`recoverDanglingHeads`): symmetric to tail recovery for branches that don't reach the source kmer.
- `minDanglingBranchLength = 4`: minimum branch length to attempt recovery.
- `recoverAllDanglingBranches = false` by default: only recovers branches with sufficient evidence (not all possible dangling branches).

Without pruning, noisy low-multiplicity paths compete with real haplotype paths in `find_best_haplotypes`. Without dangling recovery, reads that span an indel but end before the kmer sink are silently discarded, making the indel haplotype invisible.

Fix: implement edge pruning with configurable `min_prune_factor` (default 2). Implement dangling tail/head recovery. Both are in `AbstractReadThreadingGraph.java:400,427` and are well-defined graph operations.

#### Early stop on first non-reference graph does not check graph adequacy

`assemble_haplotypes` (haplotype_caller.rs lines 3781–3784):

```rust
if found_nonref {
    break;
}
```

Java also stops early, but only after checking that the graph is adequate and not suspect (no cycles, no topology errors, passes haplotype count and quality checks). The Rust early stop fires on any non-ref path, including graphs with cycles or very low evidence. Java expands k-mers starting from `max(kmerSizes) + 10` (= 35 for the default `[10,25]`) for up to `MAX_KMER_ITERATIONS_TO_ATTEMPT = 6` additional attempts (verified: `ReadThreadingAssembler.java:44-45,563-570`) when the graph has cycles, fails to find paths, or produces suspect haplotypes. The current Rust behavior can lock in a bad graph at k=10 and skip the k=25 graph that would have found the correct haplotype.

Fix: detect graph cycles before early-stop. If cycles are present or no non-reference haplotypes can be traced from source to sink, continue to the next k-mer size instead of breaking. Add k-mer expansion starting at `max(kmerSizes) + 10` for up to 6 additional attempts, matching Java's constants.

### `src/rust/src/pair_hmm.rs`

The PairHMM core algorithm is structurally correct. The initial deletion matrix initialization (`1.0 / hap_len` distributed along row 0) and the tristate correction (`TRISTATE_CORRECTION = 3.0`) match Java. The fallback values `45` for ins/del quals and `10` for GCP are correct — Java uses the same defaults (`ReadUtils.DEFAULT_INSERTION_DELETION_QUAL = 45` at `ReadUtils.java:44`, `gcpHMM = 10` at `LikelihoodEngineArgumentCollection.java:63`). The real gap is that Rust does not read the `BI`/`BD` BAM tags that BQSR writes, as described in the haplotype_caller.rs section above.

The other missing piece is dynamic read disqualification. Java's `PairHMMLikelihoodCalculationEngine` calls `filterPoorlyModeledEvidence` (line 201) to remove reads whose best haplotype likelihood is too low relative to the expected error rate. Rust has no equivalent filter. This allows reads that have very poor alignment to all haplotypes to contribute noise to the genotype likelihoods.

### `src/rust/src/smith_waterman.rs`

The Smith-Waterman implementation covers the four overhang strategies used by GATK (SoftClip, Ignore, Indel, LeadingIndel). The scoring parameters (match=1, mismatch=-4, gap_open=-6, gap_extend=-1) match Java's `ORIGINAL_DEFAULT` parameters. The `find_subsequence` uses reverse search (lastIndexOf semantics) which matches Java. No critical issues found here; this module is the most faithful to Java of all the Rust components.

### Updated recommended implementation order

Based on Java source verification above, the three changes with the best estimated sensitivity-per-effort ratio are:

1. **Raise active region padding to 100** (`ACTIVE_REGION_PADDING = 100` at haplotype_caller.rs:33). One constant change, verified against `AssemblyRegionArgumentCollection.java:24`. Expected to recover events from the 4,587 spatially isolated GATK-private calls by including reads in the assembly that currently fall outside the padded window.

2. **Raise haplotype path cap to 128** (change `10` to `128` in `assemble_haplotypes` call at hc.rs:3744). One constant change, verified against `ReadThreadingAssemblerArgumentCollection.java:112`. Expected to recover multi-event haplotypes that are currently cut off.

3. **Include soft-clipped bases in assembly reads** (add the soft-clip case in the assembly read extraction loop to mirror the existing `clip_read_for_evidence` logic). Small code change, verified against `AssemblyBasedCallerUtils.java:141-142`. Expected to improve indel sensitivity at exon boundaries.

These three can be done and measured in a single selected-region replay run before touching the more complex allele marginalization or graph pruning work. After measuring their effect on the GATK-private count, proceed with:

4. **Replace max-in-log-space with log-sum-exp marginalization** for allele likelihoods in `scan_call_window` (haplotype_caller.rs:1027–1039). Verified as the correct Java behavior in `HaplotypeCallerGenotypingEngine.java:192`. This is the core genotyping fix for the 774 `1/1→0/1` shared GT errors.

5. **Fix replay path to use assembly events** so the selected-region oracle is measuring the same algorithm as production.

6. **Read BI/BD BAM tags for PairHMM ins/del quals** to match Java's `ReadUtils.getBaseInsertionQualities` / `getBaseDeletionQualities` behavior.

7. **Graph pruning (minPruneFactor=2) and dangling-end recovery** once the above are measured and stable. Verified in `ReadThreadingAssembler.java:688-689` and `ReadThreadingAssemblerArgumentCollection.java:92,125`.
