# Step-script real data validation

## Aim

Validate the new standalone step scripts on real `SRR949115` baseline artifacts, without rerunning the whole pipeline.

## What was tested

Used existing baseline outputs from:

- `/XCLabServer002_fastIO/gatk-faster-rnaseq/SRR949115_broad_hg38_run2/baseline`

Tested scripts:

- `step_build_merged_exon_bed.py`
- `step_bed_to_interval_list.py`
- `step_split_hc_intervals.py`
- `step_haplotype_caller.py`

## Results

### 1. Merged BED generation works

Command succeeded:

- `step_build_merged_exon_bed.py --input-bed .../exons.bed --output-bed .../reference/exons.merged.bed`

Observed reduction:

- raw exon BED rows: `1,668,627`
- merged exon BED rows: `328,579`
- raw exon BED size: `38M`
- merged exon BED size: `7.5M`

So the merge step is effective and cheap enough to keep.

### 2. BedToIntervalList on merged BED is still expensive

Command started correctly:

- `step_bed_to_interval_list.py --input-bed .../exons.merged.bed --output-interval-list .../exons.merged.interval_list`

But it did not finish within the manual validation window and was interrupted.

Conclusion:

- reducing BED row count alone does not remove the one-time interval-list preparation cost;
- this conversion step is itself worth benchmarking and possibly replacing later.

### 3. SplitIntervals on the full exon interval list is also expensive

Command started correctly:

- `step_split_hc_intervals.py --input-interval-list .../exons.interval_list --scatter-count 8 --output-dir .../raw_hc_scatter`

But it also did not finish within the manual validation window and was interrupted.

Conclusion:

- shard preparation over a highly fragmented exon interval list is not cheap;
- `SplitIntervals` should be treated as a measurable hotspot, not as free orchestration.

### 4. HaplotypeCaller step script starts correctly on real BAM input

Created a tiny real interval list for smoke testing:

- `/XCLabServer002_fastIO/gatk-faster-rnaseq/step_validation/tiny_exons.interval_list`

`step_haplotype_caller.py` was launched successfully against:

- baseline `SRR949115.recal.bam`
- the tiny interval list above

The process started normally but still showed substantial fixed overhead and low observed CPU utilization during the short validation window, so it was interrupted rather than left running.

Conclusion:

- the standalone caller script is wired correctly and can consume real inputs;
- even tiny interval targets still pay a meaningful fixed `HaplotypeCaller` cost;
- future scatter experiments should not assume linear scaling from interval count alone.

## Practical conclusion

The step-script refactor is functionally valid:

- steps can now be invoked independently on real artifacts;
- whole-pipeline orchestration is no longer required for single-step experiments.

The first real-data validation also exposed two concrete optimization targets beyond the original core tools:

1. interval-list preparation from BED
2. interval scattering over fragmented exon targets

These should be measured explicitly in later benchmarks instead of being treated as negligible setup.
