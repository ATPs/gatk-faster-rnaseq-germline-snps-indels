# HaplotypeCaller active-region span diagnostics

## Aim

Expose enough replay-region metadata to evaluate active-region padding decisions from data instead of from assumptions.

## Source check

Local source verification used:

- active-region discovery and padding in `src/rust/src/haplotype_caller.rs`
- replay active-region TSV writer in `src/rust/src/haplotype_caller.rs`

Confirmed findings:

- `discover_active_regions(...)` first identifies unpadded active spans.
- It then expands those spans by `ACTIVE_REGION_PADDING` and returns the padded intervals used for fetching reads and local assembly.
- Replay output previously only wrote the padded `start`/`end` span, which made it hard to tell whether a locus was near the underlying active span edge or only near the padded edge.

## Code changes

Changed file:

- `src/rust/src/haplotype_caller.rs`

Implementation:

- Added `ActiveRegionSpan { active, padded }` so discovery preserves both intervals.
- Added `pad_active_region(...)` as a small helper for the padding/clipping step.
- Switched production and replay call paths to consume the new discovery result.
- Extended `ReplayActiveRegionRow` and `replay.active_regions.tsv` with:
  - `active_start`
  - `active_end`
  - `padded_start`
  - `padded_end`
- Added a focused unit test that verifies padding clips to the fetch window while preserving the original active span.

## Conclusion

This is a diagnostics improvement, not a padding-policy change.

The immediate value is that replay output can now show whether sampled Java-only loci are near:

- the raw active span edge;
- the padded assembly span edge;
- or neither.

That gives the next padding decision a direct measurement path in the current caller.
