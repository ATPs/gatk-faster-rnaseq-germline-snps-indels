# Concrete Plan for Rust HaplotypeCaller Parity Work

## Purpose

The previous version of this note was too confident. It mixed real gaps, stale code observations, and unverified root-cause explanations. This file should guide the next improvement rounds by making every change evidence-driven:

1. define a stable "good variant" comparison set;
2. compare Java GATK and Rust on exact allele keys;
3. randomly sample Java-only and Rust-only good variants;
4. classify why each sampled variant differs;
5. update the Rust implementation for the dominant, verified failure modes;
6. rerun the same comparison and repeat until the remaining difference is small and not dominated by one fixable cause.

Java GATK is the parity oracle for this project. It is not a biological truth set.

## Golden Good-Variant Set

The original draft used `INFO/AC >= 3`. That is impossible here after allele splitting because this is a single-sample diploid VCF, so `INFO/AC` is at most `2`.

- `FILTER == PASS`
- `FORMAT/DP[0] >= 10`
- `FORMAT/AD[0:1] >= 3`

This keeps the comparison allele-level while enforcing both minimum genotype depth and minimum ALT read support.

## Round13, Round14, and Current Round15 Change

The next implemented change is now in the current source tree:

- haplotype-to-reference alignment in `assemble_haplotypes(...)` no longer uses the generic default Smith-Waterman parameters;
- it now uses Java GATK's haplotype-to-reference scoring constants (`NEW_SW_PARAMETERS`: `200,-150,-260,-11`);
- non-reference assembled haplotypes are skipped if the path-to-reference alignment still contains `S` operators or starts at a positive reference offset;
- non-ACGT assembled haplotypes are skipped before they enter the assembled haplotype set.

Why this change was selected:

- local Java source shows HaplotypeCaller path-to-reference alignment uses `CigarUtils.calculateCigar(..., SWOverhangStrategy.SOFTCLIP)` with `NEW_SW_PARAMETERS`, and treats retained soft clips as an alignment failure rather than a valid emitted haplotype;
- the focused Rust replay at `chr1:27549213-27549413` was still carrying many long soft-clipped assembled haplotypes and emitting the Rust-only family:

## Sampling Protocol

For each round, sample exactly 20 Rust-only good variants and 20 Java-only good variants from `java_vs_rust.good.details.tsv`.

With Java as `A` and Rust as `B`:

- Java-only variants have category `a_private`.
- Rust-only variants have category `b_private`.

```bash
DETAILS="${GOOD_DIR}/java_vs_rust.good.details.tsv"

awk -F'\t' 'NR > 1 && $1 == "b_private"' "${DETAILS}" | shuf -n 20 > "${GOOD_DIR}/roundN.rust_only.sample20.tsv"
awk -F'\t' 'NR > 1 && $1 == "a_private"' "${DETAILS}" | shuf -n 20 > "${GOOD_DIR}/roundN.java_only.sample20.tsv"
```

Inspect one variant at a time. For every sampled variant, answer:

- Does the other caller have the exact allele in raw VCF but fail `PASS`, `FORMAT/DP[0]>=10`, or `FORMAT/AD[0:1]>=3`?
- Does the other caller have a nearby representation within 100 bp that suggests normalization or allele decomposition differences?
- Does the other caller have read evidence but fail to create the candidate event?
- Did candidate generation happen, but PairHMM/genotyping/annotations pushed it below the good-variant threshold?
- Is this a multi-allelic or overlapping-event case where Rust emits separate biallelic records but Java merges alleles?

Use this classification table for every sampled variant:

| round | side | chrom | pos | ref | alt | type | other raw exact? | other nearby? | other FILTER/DP/ADalt | likely failing stage | evidence checked | proposed Rust change |
|---|---|---:|---|---|---|---|---|---|---|---|---|---|

Recommended `likely failing stage` values:

- `raw_absent_candidate_generation`
- `raw_absent_assembly_haplotype`
- `raw_present_filter_qd_fs`
- `raw_present_dp_ad`
- `shared_key_gt_or_pl`
- `normalization_or_multiallelic`
- `read_finalization_or_downsampling`
- `unclear_needs_replay`

## Variant Inspection Workflow

For a Rust-only good variant, first find why Java excluded it:

1. Query Java raw and filtered VCFs at the exact allele and within +/-100 bp.
2. If Java raw contains the exact allele, record Java `FILTER`, `DP`, `AD`, `QD`, `FS`, `GT`, `PL`, and explain which good-set condition failed.
3. If Java raw has only a nearby equivalent-looking allele, classify as normalization/decomposition.
4. If Java raw lacks the allele, run selected-region Java oracle and Rust replay around that locus to compare active region, haplotype, PairHMM, and genotype evidence.

For a Java-only good variant, mirror the same process on Rust:

1. Query Rust raw and filtered VCFs at the exact allele and within +/-100 bp.
2. If Rust raw contains the exact allele but it does not enter the Rust-good set, fix annotations, genotype, or filtering logic.
3. If Rust raw lacks the allele, determine whether Rust missed active-region discovery, read finalization, assembly haplotype creation, PairHMM likelihood, or final genotyping.

Per-locus region window:

```bash
CHROM=chr1
POS=123456
START=$((POS > 150 ? POS - 150 : 1))
END=$((POS + 150))

bcftools view -r "${CHROM}:${START}-${END}" "${JAVA_FILTERED}" > "${GOOD_DIR}/inspect.java.${CHROM}_${POS}.filtered.vcf"
bcftools view -r "${CHROM}:${START}-${END}" "${RUST_FILTERED}" > "${GOOD_DIR}/inspect.rust.${CHROM}_${POS}.filtered.vcf"
bcftools view -r "${CHROM}:${START}-${END}" "${JAVA_RAW}" > "${GOOD_DIR}/inspect.java.${CHROM}_${POS}.raw.vcf"
bcftools view -r "${CHROM}:${START}-${END}" "${RUST_RAW}" > "${GOOD_DIR}/inspect.rust.${CHROM}_${POS}.raw.vcf"
```

For deeper stage debugging, create a small interval list from sampled loci, then run:

- `src/rust/bin/rust_hc_run_gatk_oracle`

- `src/rust/bin/rust_hc_active_region_diff`
- `src/rust/bin/rust_hc_assembly_diff`

- `src/rust/bin/rust_hc_genotype_diff`

Use the commands already documented in `build.md` as the template.

## 2026-06-18 Round27 to Round30 Update: Read `N` Handling and Strong SNP Rescue

### Java `N` Handling Check

Local Java source confirms that `N` has two different meanings in two different stages:

- Assembly graph input:
  - `AbstractReadThreadingGraph.addRead()` splits reads into maximal usable segments.
  - `baseIsUsableForAssembly()` rejects read bases equal to `N` and bases below the assembly base-quality threshold.
  - Therefore read base `N` should not be threaded into the assembly graph as an ordinary base.
- PairHMM scoring:
  - `LoglessPairHMM` and `Log10PairHMM` treat either read base `N` or haplotype base `N` as a match in the prior matrix.
  - Rust PairHMM already had equivalent `rb == hb || rb == b'N' || hb == b'N'` behavior.
- Active-region discovery:
  - Java `ReferenceConfidenceModel.isAltBeforeAssembly()` treats any pileup base different from the reference as non-reference evidence. This includes read base `N`.
  - Java `ReadPileup.getBaseCounts()` still ignores `N` for concrete A/C/G/T allele counts.

Rust has now been changed to follow this split more closely:

- Assembly uses Java-like read segments that break on read base `N` or low base quality.
- PairHMM still scores the full prepared read sequence and keeps the existing `N` wildcard behavior.
- SNP pileup collection now keeps separate `active_observations`, so non-ACGT bases can affect active-region discovery without entering A/C/G/T allele counts, AD, or final SNP genotype evidence.
