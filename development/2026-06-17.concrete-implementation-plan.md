# Concrete Plan for Rust HaplotypeCaller Parity Work

## Purpose

The previous version of this note was too confident. It mixed real gaps, stale code observations, and unverified root-cause explanations. This file should guide the next improvement rounds by making every change evidence-driven:

1. define a stable "good variant" comparison set;
2. compare Java GATK and Rust on exact allele keys;
3. randomly sample Java-only and Rust-only good variants;
4. classify why each sampled variant differs;
5. update the Rust implementation for the dominant, verified failure modes;
6. rerun the same comparison and repeat until the remaining difference is small and not dominated by one fixable cause.

Java GATK is the parity oracle for this project. It is not a biological truth set.

## Golden Good-Variant Set

For each Java and Rust result, define the comparison set as:

- `FILTER == PASS`
- `INFO/DP >= 10`
- `INFO/AC >= 3`

This removes weak low-depth calls from both sides before judging caller parity.

## Sampling Protocol

For each round, sample exactly 20 Rust-only good variants and 20 Java-only good variants from `java_vs_rust.good.details.tsv`.

With Java as `A` and Rust as `B`:

- Java-only variants have category `a_private`.
- Rust-only variants have category `b_private`.

```bash
DETAILS="${GOOD_DIR}/java_vs_rust.good.details.tsv"

awk -F'\t' 'NR > 1 && $1 == "b_private"' "${DETAILS}" | shuf -n 20 > "${GOOD_DIR}/roundN.rust_only.sample20.tsv"
awk -F'\t' 'NR > 1 && $1 == "a_private"' "${DETAILS}" | shuf -n 20 > "${GOOD_DIR}/roundN.java_only.sample20.tsv"
```

Inspect one variant at a time. For every sampled variant, answer:

- Does the other caller have the exact allele in raw VCF but fail `PASS`, `DP>=10`, or `AC>=3`?
- Does the other caller have a nearby representation within 100 bp that suggests normalization or allele decomposition differences?
- Does the other caller have read evidence but fail to create the candidate event?
- Did candidate generation happen, but PairHMM/genotyping/annotations pushed it below the good-variant threshold?
- Is this a multi-allelic or overlapping-event case where Rust emits separate biallelic records but Java merges alleles?

Use this classification table for every sampled variant:

| round | side | chrom | pos | ref | alt | type | other raw exact? | other nearby? | other FILTER/DP/AC | likely failing stage | evidence checked | proposed Rust change |
|---|---|---:|---|---|---|---|---|---|---|---|---|---|

Recommended `likely failing stage` values:

- `raw_absent_candidate_generation`
- `raw_absent_assembly_haplotype`
- `raw_present_filter_qd_fs`
- `raw_present_dp_ac`
- `shared_key_gt_or_pl`
- `normalization_or_multiallelic`
- `read_finalization_or_downsampling`
- `unclear_needs_replay`

## Variant Inspection Workflow

For a Rust-only good variant, first find why Java excluded it:

1. Query Java raw and filtered VCFs at the exact allele and within +/-100 bp.
2. If Java raw contains the exact allele, record Java `FILTER`, `DP`, `AC`, `QD`, `FS`, `GT`, `AD`, `PL`, and explain which good-set condition failed.
3. If Java raw has only a nearby equivalent-looking allele, classify as normalization/decomposition.
4. If Java raw lacks the allele, run selected-region Java oracle and Rust replay around that locus to compare active region, haplotype, PairHMM, and genotype evidence.

For a Java-only good variant, mirror the same process on Rust:

1. Query Rust raw and filtered VCFs at the exact allele and within +/-100 bp.
2. If Rust raw contains the exact allele but it does not enter the Rust-good set, fix annotations, genotype, or filtering logic.
3. If Rust raw lacks the allele, determine whether Rust missed active-region discovery, read finalization, assembly haplotype creation, PairHMM likelihood, or final genotyping.

Per-locus region window:

```bash
CHROM=chr1
POS=123456
START=$((POS > 150 ? POS - 150 : 1))
END=$((POS + 150))

bcftools view -r "${CHROM}:${START}-${END}" "${JAVA_FILTERED}" > "${GOOD_DIR}/inspect.java.${CHROM}_${POS}.filtered.vcf"
bcftools view -r "${CHROM}:${START}-${END}" "${RUST_FILTERED}" > "${GOOD_DIR}/inspect.rust.${CHROM}_${POS}.filtered.vcf"
bcftools view -r "${CHROM}:${START}-${END}" "${JAVA_RAW}" > "${GOOD_DIR}/inspect.java.${CHROM}_${POS}.raw.vcf"
bcftools view -r "${CHROM}:${START}-${END}" "${RUST_RAW}" > "${GOOD_DIR}/inspect.rust.${CHROM}_${POS}.raw.vcf"
```

## Current Implementation Priorities

Do not implement all possible GATK differences at once. Pick the next change from the dominant class in the 40 sampled variants.

### 1. Fix Production Read Subset Consistency

Current status:

- `scan_call_window` computes a Java-like per-start `use_for_assembly` flag with cap 50.
- It then ignores that flag and uses `read_bases_list.iter().take(500)` for assembly.
- `scan_replay_window` does use the per-start flag.

Why this matters:

- Production and replay inspect different read subsets.
- Assembly can be biased by coordinate-sorted input order.
- PairHMM scores all reads while assembly sees only the first 500 reads, making candidate generation and genotyping inconsistent.

Implementation target:

- Replace production `take(500)` with the same `read_use_for_assembly_list` filtering used by replay.
- Keep PairHMM scoring on the intended Java-like finalized read set.
- Record assembly-read counts in replay so sampled variants can show whether a locus was read-subset limited.

Verification:

- Unit test that per-start cap keeps at most 50 reads per alignment start.
- Selected-region replay and production should report the same assembled haplotypes/events for the same locus.

### 2. Fix Allele Likelihood Marginalization

Current status:

- Rust currently takes the best haplotype likelihood in each allele class.
- Java maps alleles to all supporting haplotypes and marginalizes with `AlleleLikelihoods.marginalize(alleleMapper)`.

Implementation target:

- Replace max-in-log-space with log10-sum-exp over all haplotypes supporting REF and ALT.
- Fix the misleading comment that says max matches GATK.
- Add a small unit test where two ALT haplotypes jointly produce a higher ALT allele likelihood than either haplotype alone.

Verification:

- Shared genotype differences should decrease, especially Java `1/1` vs Rust `0/1`.
- Good-set Rust-only calls should not expand substantially.

### 3. Fix Base-Quality Threshold Handling

Current status:

- Rust uses `quals[i].min(mapq).max(18)` in both production and replay.
- Java caps base quality by mapping quality, then changes below-threshold qualities to the minimum usable quality rather than raising them to 18.

Implementation target:

- Replace `.max(18)` with Java-like threshold behavior.
- Make the threshold a named constant tied to the PairHMM base-quality score threshold.

Verification:

- Sampled Rust-only variants caused by low-quality evidence should either disappear from the good set or match Java filters better.
- Confirm no large loss of Java-good shared calls.

### 4. Fix Replay Genotype Emission

Current status:

- Replay emits assembled event rows from `valid_events`.
- Replay genotype rows are generated from `output.variants`.
- `scan_replay_window` currently extends `output.variants` with `region_events`, but `region_events` is not populated from the assembled/genotyped calls.

Implementation target:

- Populate replay genotype rows from the same final calls that production would emit for the region.
- Ensure selected-region genotype diffs compare Java against the actual Rust production path.

Verification:

- `HaplotypeReplayStats.genotype_rows` should match the number of replay final variant rows.
- `rust_hc_genotype_diff` should be meaningful on selected regions.

### 5. Active-Region Padding 50 to 100

Current status:

- Rust padding is 50.
- Java default padding is 100.

Implementation target:

- Change `ACTIVE_REGION_PADDING` to 100 only after the sampling shows Java-only good variants are often absent from Rust raw and near active-region boundaries.
- Add replay columns for active span and padded span so the effect is measurable.

Verification:

- Java-only good variants near current region boundaries should decrease.
- Runtime and Rust-only good variants should not increase unacceptably.

### 6. Graph Adequacy, K-mer Expansion, and Dangling Recovery

Current status:

- Basic pruning and max haplotype cap are already present.
- Rust still lacks Java-like graph adequacy checks, cycle-triggered k-mer expansion, and dangling head/tail recovery.

Implementation target:

- Add cycle detection and continue to larger k-mers when the current graph is cyclic or suspect.
- Add Java-like dangling tail/head recovery only after sampled Java-only variants show haplotypes are being lost at graph ends.

Verification:

- Improvements should be visible in Java-only indel/complex samples, not just total sensitivity.

### 7. Multi-allelic and Overlapping Event Genotyping

Current status:

- Rust still genotypes events mostly as independent biallelic calls.
- Java merges overlapping events, builds an allele mapper, marginalizes haplotypes to alleles, and genotypes all alleles at the locus together.

Implementation target:

- Group overlapping candidate variants.
- Emit multi-ALT records where Java does.
- Compute diploid genotype likelihoods for all allele pairs.

Verification:

- Sampled `normalization_or_multiallelic` and shared genotype differences should decrease.

## Round Acceptance Rules

After each Rust code change:

1. Run focused unit tests and release build:

```bash
cd /data/p/gatk/gatk-faster-rnaseq-germline-snps-indels
/data/p/sys/rust/1.96.0/bin/cargo fmt --manifest-path src/rust/Cargo.toml --check
/data/p/sys/rust/1.96.0/bin/cargo test --manifest-path src/rust/Cargo.toml --lib
/data/p/sys/rust/1.96.0/bin/cargo test --manifest-path src/rust/Cargo.toml --bins
/data/p/sys/rust/1.96.0/bin/cargo build --release --manifest-path src/rust/Cargo.toml --bins
```

2. Rerun selected-region comparison on the sampled loci.
3. If selected regions improve without creating many new Rust-only good calls, rerun full module 6 in `tmux`.
4. Recreate Java/Rust good VCFs with `PASS && DP>=10 && AC>=3`.
5. Compare exact allele keys and record the new summary.
6. Randomly sample another 20 Rust-only and 20 Java-only good variants.
7. Decide the next code change from the new sample, not from old assumptions.

A change is acceptable only if it improves at least one of these without a large regression in the others:

- fewer Java-only good variants;
- fewer Rust-only good variants;
- fewer shared genotype differences;
- clearer replay diagnostics for the remaining variants.

Stop iterating when both private good-variant sets are small enough for the project goal and the latest random sample no longer points to a common Rust implementation defect.
