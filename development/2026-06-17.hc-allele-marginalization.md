# HaplotypeCaller allele-likelihood marginalization parity fix

## Aim

Bring Rust HaplotypeCaller genotyping closer to Java by marginalizing read likelihoods across all haplotypes that support an allele, instead of keeping only the single best haplotype likelihood per allele class.

## Source check

Local source verification used:

- Rust genotyping loop in `src/rust/src/haplotype_caller.rs`
- Java genotyping flow in `/data/p/gatk/gatk/src/main/java/org/broadinstitute/hellbender/tools/walkers/haplotypecaller/HaplotypeCallerGenotypingEngine.java`

Confirmed findings:

- Rust already partitioned per-read haplotype likelihoods into REF-supporting and ALT-supporting sets.
- The old Rust code then collapsed each set with `max`, which throws away cumulative support from multiple moderately likely haplotypes.
- Java builds an `alleleMapper` and calls `readLikelihoods.marginalize(alleleMapper)` before downstream genotyping and annotations.

Conclusion from source: the Rust caller had a real genotyping mismatch here, and the old comment claiming max matched GATK was wrong.

## Code changes

Changed file:

- `src/rust/src/haplotype_caller.rs`

Implementation:

- Added `marginalize_allele_likelihoods(values: &[f64]) -> f64`.
- Implemented that helper as:
  - `NEG_INFINITY` for empty support;
  - `log10_sum_exp(values)` otherwise.
- Replaced the per-read REF/ALT `max` collapse in the assembled-event genotyping loop with the shared marginalization helper.
- Added unit tests that verify:
  - two supporting haplotypes marginalize to a stronger allele likelihood than either haplotype alone;
  - empty support returns `NEG_INFINITY`.

## Conclusion

This fixes one confirmed Java/Rust genotyping mismatch in the current Rust caller.

The next step is not another local refactor. It is to rerun the Java-vs-Rust good-set comparison with the current binary and see whether:

- shared genotype differences decrease;
- Java-only good variants shrink;
- Rust-only good variants do not grow materially.
