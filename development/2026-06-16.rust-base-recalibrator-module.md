# Rust BaseRecalibrator module validation

## Aim

Validate module 4 from `2026-06-16.optimization-roadmap.md`: a Rust replacement for GATK `BaseRecalibrator` on the real `SRR949115` RNA-seq preprocessing BAM.

The benchmark input is the same split BAM used by the baseline run:

- BAM: `/XCLabServer002_fastIO/gatk-faster-rnaseq/SRR949115_broad_hg38_run2/baseline/SRR949115.split.bam`
- BAI: `/XCLabServer002_fastIO/gatk-faster-rnaseq/SRR949115_broad_hg38_run2/baseline/SRR949115.split.bai`
- Reference: `/data1/pub/gatk/broad_hg38/Homo_sapiens_assembly38.fasta`
- Known sites:
  - `/data1/pub/gatk/broad_hg38/Homo_sapiens_assembly38.dbsnp138.vcf.gz`
  - `/data1/pub/gatk/broad_hg38/Homo_sapiens_assembly38.known_indels.vcf.gz`
  - `/data1/pub/gatk/broad_hg38/Mills_and_1000G_gold_standard.indels.hg38.vcf.gz`

## Implementation status

Implemented `rust_base_recalibrator` as an opt-in backend for `src/step_base_recalibrator.py`.

Current scope:

- mismatch event table generation;
- read group, reported quality, cycle, and context covariates;
- `--use-original-qualities`;
- known-sites masking from indexed VCFs;
- GATK report-compatible output tables;
- indexed BAM parallel traversal with `--threads` and `--region-bases`.

Not yet validated as a default replacement:

- downstream VCF concordance after applying the Rust table;
- exact GATK behavior on every alternate contig;
- insertion/deletion event model parity.

## Thread determinism check

Input slice:

- `/XCLabServer002_fastIO/gatk-faster-rnaseq/module4_thread_check_20260616/chr1_1m_1p1m.split.bam`

Rust was run on the same slice with `--threads 1`, `--threads 8`, and `--threads 40`, forcing small chunks with `--region-bases 25000`.

All three recal tables had identical SHA-256:

- `22ee3e239aa2217fb8cda934e0165b1fa4ab08bb188896c855f51f50ad5eda51`

Small-slice timings:

| mode | wall time | max RSS |
|---|---:|---:|
| Rust 1 thread | 0.49 s | 82,540 KB |
| Rust 8 threads | 0.63 s | 640,560 KB |
| Rust 40 threads | 2.21 s | 3,141,108 KB |

For a tiny slice, thread startup and per-thread readers dominate, so this check is for correctness, not speed.

## Full real-data speed benchmark

Output directory:

- `/XCLabServer002_fastIO/gatk-faster-rnaseq/module4_base_recalibrator_speed_threads40_20260616`

Commands were run under zsh with conda base activated. Other long-running GATK jobs were active on the host, so absolute wall times should be treated as busy-host measurements.

### Runtime

| backend | threads | wall time | CPU | max RSS | exit |
|---|---:|---:|---:|---:|---:|
| GATK BaseRecalibrator 4.6.2.0 | default | 391.36 s | 108% | 1,290,304 KB | 0 |
| Rust BaseRecalibrator | 40 | 50.80 s | 2525% | 3,523,812 KB | 0 |

Speedup versus this same-run GATK measurement:

- `391.36 / 50.80 = 7.70x`

Speedup versus the earlier baseline timing from `2026-06-16.baseline-sambamba-benchmark.md`:

- `365.846 / 50.80 = 7.20x`

The Rust process was observed at about 3800% CPU early in the run, and averaged 2525% over the whole command.

### Recal table summary

| backend | RecalTable0 observations | RecalTable0 errors | RecalTable1 rows | RecalTable2 rows |
|---|---:|---:|---:|---:|
| GATK | 1,165,001,001 | 3,824,157 | 36 | 7,207 |
| Rust 40 | 1,164,981,745 | 3,827,365 | 36 | 7,207 |

Differences:

- observations: `-19,256` (`-0.00165%`);
- errors: `+3,208` (`+0.0839%`).

## Caveats

The Rust run emitted 1,254 warnings for 423 alternate/HLA contigs where htslib could resolve the contig name in a known-sites header but could not seek that contig in the tabix index. The Rust implementation currently treats those contigs as having no known-sites records and continues. This avoids failing the run, but it likely explains part of the remaining count difference versus GATK.

Before adopting this as a default backend, run:

- GATK `ApplyBQSR` with the Rust table;
- downstream `HaplotypeCaller` and filtration;
- VCF count and `bcftools isec` comparison against the baseline GATK table output.

## Current conclusion

On the real `SRR949115.split.bam`, the 40-thread Rust `BaseRecalibrator` is much faster than GATK for table generation:

- `50.80 s` Rust 40-thread wall time versus `391.36 s` GATK wall time;
- `7.70x` faster on this busy-host same-run comparison;
- recal table row shape matches GATK, with a small full-data count difference that still needs downstream VCF validation.
