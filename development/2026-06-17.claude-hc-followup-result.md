# HaplotypeCaller follow-up fix: read-finalization clipping and active-locus detection

## Aim

Fix the partial/incomplete active-region edit left by a prior run that exited mid-edit. Recover the useful algorithmic slice (GATK-like read clipping and active-locus detection), remove the non-coherent duplicate evidence-collection path, and leave the repo compiling clean.

## What was broken in the partial edit

Three categories of compile errors in `src/rust/src/haplotype_caller.rs`:

1. **Cigar iteration** — `clip_read_for_evidence` and `collect_region_evidence` used `for op in record.cigar()` directly. `CigarStringView` does not implement `IntoIterator` in rust-htslib 1.0.0; it requires `.iter()`.

2. **Type dereference errors** — Pattern-matching on `Cigar::Match(l)` in the context of `.iter()` yields `l: &u32`, so `*l as usize` was correct (deref ref then cast). But `Cigar::Del(d) => *d as u32` was wrong in context: `d: &u32` gives `*d: u32`, and the arm returned `u32` while the `let len` binding inferred `usize` from other arms (type mismatch).

3. **Mixed usize/u32 arithmetic** — `Cigar::Ins(ins_len)` bound `ins_len: &u32`, but it was compared to `MAX_BOOTSTRAP_INDEL_LEN as usize` (type mismatch) and used in range `qpos..qpos + ins_len` (usize + &u32).

Additionally, `collect_region_evidence` re-implemented htslib pileup logic by walking BAM CIGAR manually. This duplicated the existing evidence functions without adding new behavior and was not wired into `scan_call_window` or `scan_replay_window`.

## Java GATK methods inspected

- **`AssemblyBasedCallerUtils.java`** — `finalizeRegion()` (line 121):
  - Hard-clips soft-clipped bases via `ReadClipper.hardClipSoftClippedBases()` when `dontUseSoftClippedBases` is true.
  - Hard-clips low-quality tail ends via `ReadClipper.hardClipLowQualEnds()` with `minTailQuality = minBaseQualityScore - 1`.
  - Removes reads shorter than `MINIMUM_READ_LENGTH_AFTER_TRIMMING` (10 bp) after trimming.

- **`HaplotypeCallerEngine.java`** — `isActive()`:
  - Uses `ReferenceConfidenceModel.calcGenotypeLikelihoodsOfRefVsAny()` with a low discovery confidence threshold (4.0 phred) to flag active loci.

- **`ReferenceConfidenceModel.java`** — `calcGenotypeLikelihoodsOfRefVsAny()`:
  - Computes ref-vs-any likelihoods treating all non-reference bases as alt evidence.

## Final Rust algorithm change (Option 2: simplified but useful)

1. **Fixed `clip_read_for_evidence`**: Corrected `cigar.iter()` iteration and type handling. The function now correctly computes the valid read coordinate span after:
   - Hard-clipping soft-clipped bases (when `--dont-use-soft-clipped-bases`)
   - Trimming low-quality tail bases (quality < `PIPELINE_MIN_TAIL_QUALITY = 9`)
   - Removing reads shorter than 10 bp after trimming
   Mirrors GATK's `ReadClipper.hardClipSoftClippedBases()` + `ReadClipper.hardClipLowQualEnds()` from `finalizeRegion()`.

2. **Integrated clipping into pileup evidence collection**: Modified `pileup_snp_evidence_with_rows` and `pileup_indel_evidence_with_rows` to accept `min_tail_quality` and `dont_use_soft_clipped_bases` parameters. Before collecting each alignment's base/indel observation, the functions now check that the qpos falls within the clipped read range. Reads that are fully clipped away are excluded from evidence. Both `scan_call_window` and `scan_replay_window` pass these parameters through.

3. **Used `is_active_locus` for replay active-locus detection**: Changed `scan_replay_window` from the simple `best_alt_count > 0` heuristic to the GATK-like `is_active_locus()` function, which computes ref-vs-any diploid genotype likelihoods with a discovery confidence threshold of 4.0 (matching GATK's `isActive()`).

4. **Added `PIPELINE_MIN_TAIL_QUALITY` constant**: Set to `PIPELINE_MIN_BASEQ - 1 = 9`, matching GATK's non-error-correction tail quality threshold.

5. **Removed `collect_region_evidence`**: This function duplicated htslib pileup logic, had compile errors, and was not integrated into any call path. Its approach (manual CIGAR walking per BAM record) is fundamentally different from the pileup-based design. The function was ~220 lines of duplicated evidence-collection logic.

6. **Removed `ActiveRegion` struct and `group_active_regions`**: These were not wired into any call path. They represent scaffolding for future assembly-based active-region processing but are currently dead code. They should be reintroduced when local assembly is implemented.

## Whether this should be kept or reverted

**Keep as a small correctness slice, but do not count it as meaningful concordance progress.** The changes are:
- Bug fixes that make the partial edit compile and integrate
- Algorithmic improvements traceable to Java GATK
- No speculative or dead code remains
- The read-finalization clipping is a real improvement to evidence collection quality, but measured selected-region VCF concordance was unchanged

The removed code (`collect_region_evidence`, `ActiveRegion`, `group_active_regions`) was duplicative or scaffolding not yet wired in. Both can be reintroduced when local assembly is implemented.

## Remaining risks and next step

1. **Read clipping may reduce sensitivity slightly**: Soft-clipped bases can carry variant evidence in RNA-seq. Hard-clipping them removes this evidence. The impact should be measured via selected-region comparison.

2. **Performance of per-alignment clipping**: `clip_read_for_evidence` is called for every pileup alignment, iterating the full CIGAR each time. A caching layer could reduce overhead but adds memory. Measure before optimizing.

3. **Next recommended GATK-like step**: Implement local read-threading assembly (a minimal kmer-graph assembler) for active regions. Even a simple de Bruijn graph-based assembler that outputs reference + alternate haplotypes would substantially reduce the remaining discordance, especially for indels.
