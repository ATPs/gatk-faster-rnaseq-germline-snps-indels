# HaplotypeCaller Best-Haplotype-Pair Genotyping Analysis

## Summary

The earlier overlapping-alt-haplotype exclusion helped, but it still left many repetitive indel loci either collapsed to `0/0` or expanded into families of independently-PASS biallelic records.

A stronger focused approximation worked better:

- score diploid haplotype pairs directly from PairHMM read likelihoods;
- choose the best haplotype pair for the region;
- genotype each event from pair support instead of treating every event independently against all non-event haplotypes.

On the first `20` sampled Java-only loci, this sharply reduced the replay genotype explosion and recovered several stubborn Java-like indels.

## Why This Was Needed

After the overlapping-alt-haplotype exclusion change:

- centered zero-threshold replay still emitted `345` genotype rows across the `20` sampled Java-only regions;
- exact non-reference recovery improved only to `3/20`;
- repetitive indel families still leaked through as many independently-PASS biallelic alleles.

This matched the full round3 regression:

- Java-only good improved again (`1350 -> 1268`);
- but Rust-only good jumped badly (`3354 -> 3925`);
- most of the new Rust-only sample became `exact_absent` indel/complex calls in Java.

Representative round3 full-VCF example:

`chr4:176328436`

- Java: one allele `GTTTTTTT > G`, `PASS`, `0/1`, `AD=5,10`, `DP=15`
- Rust: a whole same-start deletion ladder, many `PASS`:
  - `GTT > G`
  - `GTTT > G`
  - `GTTTT > G`
  - `GTTTTT > G`
  - `GTTTTTT > G`
  - `GTTTTTTT > G`
  - `GTTTTTTTT > G`
  - `GTTTTTTTTT > G`

That is a same-locus genotyping failure, not just a threshold issue.

## Focused Best-Pair Replay Result

Using centered zero-threshold replay on the same `20` Java-only sampled loci:

- before best-pair change: `345` replay genotype rows
- after best-pair change: `26` replay genotype rows

Exact non-reference recovery among the sampled loci:

- before overlapping-alt exclusion: `1/20`
- after overlapping-alt exclusion: `3/20`
- after best-haplotype-pair genotyping: `6/20`

Recovered loci with Java-like single-allele outputs:

| locus | replay call after best-pair genotyping |
|---|---|
| `chr19:18280054 A>G` | `0/1`, `QUAL=2263`, `DP=359`, `AD=198,161` |
| `chr1:28499629 A>G` | `0/1`, `QUAL=119`, `DP=63`, `AD=42,21` |
| `chr18:23482513 GA>G` | `0/1`, `QUAL=52`, `DP=21`, `AD=15,6` |
| `chr3:31532901 T>TTCCTCC` | `0/1`, `QUAL=60`, `DP=12`, `AD=9,3` |
| `chr3:152463139 CT>C` | `0/1`, `QUAL=66`, `DP=25`, `AD=20,5` |
| `chr6:21596269 C>CA` | `0/1`, `QUAL=272`, `DP=55`, `AD=34,21` |

Notably, the repetitive same-position indel families at these loci collapsed to exactly one emitted allele in replay:

- `chr18:23482513`: only `GA > G`
- `chr6:21596269`: only `C > CA`
- `chr3:31532901`: only `T > TTCCTCC`
- `chr3:152463139`: only `CT > C`

## Mini Full-Caller Probe

To verify that this was not replay-only, the current release caller was run on a small interval list covering the main problematic loci:

- `chr4:176328436`
- `chr18:23482513`
- `chr6:21596269`
- `chr3:31532901`
- `chr3:152463139`

## Interpretation

This is the strongest evidence so far that:

1. the dominant remaining problem is same-locus / overlapping-event genotyping;
2. a diploid haplotype-pair view is much closer to the needed behavior than per-event independent scoring;
3. promoting best-pair genotyping into the main caller path is a justified next full-run experiment.

It is still not full Java-like multi-allelic genotyping, but it is closer to the real model and avoids the indel-family leak much better than the earlier heuristic.
