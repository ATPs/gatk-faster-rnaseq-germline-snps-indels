# VCF Concordance Gap Remediation Plan

This plan outlines the concrete steps required to bring the Rust HaplotypeCaller output into 100% concordance with the Java GATK baseline. It details the algorithmic reasons for the current remaining differences and provides a step-by-step implementation guide.

---

## 1. Dangling-End and Dangling-Head Recovery in the Graph

### The Reason:
In RNA-seq data, many reads start or end in the middle of active regions. If a read contains a variant near its end, the sequence might terminate before reaching the sink k-mer (or start after the source k-mer).
* **Current Gap**: In the current Rust graph implementation, these paths are left "dangling" (vertices with in-edges but no out-edges). Because the path-finder only traces paths that connect the `source` vertex to the `sink` vertex, these variant-bearing paths are completely ignored, causing false negatives.
* **GATK Behavior**: GATK recovers dangling ends by merging them back into the reference path (using prefix/suffix alignment) so they can successfully reach the sink.

### How to Implement:
Port GATK's dangling-end recovery from `AbstractReadThreadingGraph.java` to `src/rust/src/assembly.rs`:
1. **Identify Dangling Vertices**:
   * Identify all vertices in the graph that have in-edges but no out-edges (dangling tails), and vertices that have out-edges but no in-edges (dangling heads).
2. **Path Tracing & Alignment**:
   * Trace the path backwards/forwards from the dangling vertex to determine its length.
   * If the dangling path length is $\ge \text{minDanglingBranchLength} = 4$, extract its sequence.
3. **Reference Matching & Edge Insertion**:
   * Search for a suffix or prefix match of the dangling sequence against the reference path.
   * Connect the dangling vertex back to the matching reference vertex by adding bridging edges with appropriate multiplicities, allowing the path to flow all the way from `source` to `sink`.

## 2. Adaptive K-mer Size Expansion

### The Reason:
Active exons frequently contain short sequence repeats, homopolymers, or complex loops. At the default $k=10$ or $k=25$, these repeats generate cycles (loops) in the de Bruijn graph.
* **Current Gap**: When cycles are present, the Rust path-finder caps vertex expansions at `10,000` to avoid infinite loops, which often means the correct haplotype path is never reached. This causes a significant drop in sensitivity in repeat regions.
* **GATK Behavior**: GATK inspects the graph for cycles before finding haplotypes. If the graph is cyclic or topology-suspect, it discards the graph and re-assembles the reads using an increased k-mer size ($k_{\text{new}} = k_{\text{current}} + 10$).

### How to Implement:
Modify `assemble_haplotypes` in `src/rust/src/haplotype_caller.rs`:
1. **Cycle Detection**:
   * Implement a cycle-detection pass on `ReadThreadingGraph` (e.g., using Tarjan's strongly connected components algorithm or a simple Depth First Search path tracker).
2. **K-mer Iteration Loop**:
   * If a graph has cycles, or if no valid paths can trace from `source` to `sink`, discard the graph.
   * Re-run assembly by increasing the k-mer size by 10 (e.g., trying $k = 35, 45, 55, 65, 75, 85$).
   * Cap the fallback at `MAX_KMER_ITERATIONS_TO_ATTEMPT = 6` additional attempts (matching GATK's default parameters).

## 3. Unified Multi-Allelic Genotyping & Locus Merging

### The Reason:
Currently, the Rust caller operates event-by-event (biallelic). If a locus contains multiple overlapping variants (e.g., a multi-allelic SNP or overlapping SNP and Indel), Rust evaluates them independently.
* **Current Gap**: Rust outputs separate biallelic records. This skews the allele depth (`AD`), genotype quality (`GQ`), and quality score (`QUAL`), causing downstream filtration disagreements. GATK expects a single multi-allelic record with `GT=1/2` instead of two `0/1` records.
* **GATK Behavior**: GATK merges overlapping candidate events, constructs an allele mapper, marginalizes the read-haplotype likelihoods, and genotypes all alleles together.

### How to Implement:
Refactor the genotyping stage in `src/rust/src/haplotype_caller.rs`:
1. **Group Variants**:
   * Group candidate variants that start at the same genomic position or overlap in span.
2. **Build Allele Mapper**:
   * For $N$ alternate alleles, construct a mapper that assigns each allele to the haplotypes that support it.
3. **Multi-allelic Marginalization**:
   * Marginalize read-haplotype likelihoods into read-allele likelihoods by taking the maximum likelihood of any haplotype supporting each allele class (matching `marginalLikelihoodsDirect` in `AlleleLikelihoods.java`).
4. **Diploid Genotyping**:
   * Calculate genotype likelihoods for all $N(N+1)/2$ diploid genotype combinations (e.g., for 2 alternate alleles, evaluate `0/0`, `0/1`, `1/1`, `0/2`, `1/2`, `2/2`).
   * Emit a single unified multi-allelic VCF record with comma-separated alternate alleles, populating the unified `AD`, `PL`, `GQ`, and `QUAL` vectors.

## 4. Overlapping Mate Read Correction

### The Reason:
Since RNA-seq reads are paired-end, mates frequently overlap in the coding region.
* **Current Gap**: The Rust caller processes both mates independently, double-counting the evidence (qualities and bases) in the overlapping window, inflating likelihoods and depths.
* **GATK Behavior**: GATK clips overlapping mates or zeroes out the base qualities of one mate to prevent double-counting.

### How to Implement:
Implement GATK's `cleanOverlappingReadPairs()` in `src/rust/src/haplotype_caller.rs` during read loading:
1. **Identify Overlaps**:
   * Identify pairs of reads with matching query names that overlap in genomic coordinates.
2. **Quality Adjustment**:
   * For matching bases in the overlap, keep the base with the higher quality and set the other to quality `0`.
   * For mismatching bases, set the quality of the lower-quality base to `0` and keep the higher-quality base.
